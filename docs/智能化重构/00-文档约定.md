# 文档约定

## 1. 目标

本文档用于统一 `docs/智能化重构/` 的写法、状态、证据要求和维护方式，避免再次出现以下问题：

- 把目标态写成现状
- 把候选方案写成定论
- 把系统级顶层定义和执行域细节混在一篇文档里
- 文档之间对象命名一致，但与代码、SQL 和部署现实不一致

## 2. 基本原则

- 代码、初始化 SQL、迁移、配置和当前机器状态优先于文档
- 当前已实现、目标态建议和待决项必须分开写
- 文档不能代替设计评审、迁移验证和发布验证
- 除非代码已经落地，否则不要使用“权威结论”“最终方案”“最优解”这类表述
- 目标态完整闭环、当前最优收敛线、第二阶段增强和北极星能力必须分开写

## 3. 文档类型与状态约定

### 3.1 文档类型

当前目录文档分为六类：

- 系统级顶层定义文档
  - 回答“最终产品是什么、覆盖哪些边界”
- 系统级控制平面或实施蓝图文档
  - 回答“全局如何被统一控制、如何分阶段落地”
- 领域 `RFC` 文档
  - 回答“某一域当前现实、目标设计和落地顺序”
- 共享契约文档
  - 回答“跨域共享对象、标识、字段和兼容规则”
- 现状真相文档
  - 回答“当前代码和数据库已经是什么样”
- 施工附录文档
  - 回答“真实改哪些文件、补哪些测试、跑哪些命令、从哪里回滚”

### 3.2 文档状态

每份文档开头都应标注状态，统一使用以下枚举：

- `现状已核对`
  - 说明文档主要描述当前实现，且已按代码或 SQL 复核
- `草案`
  - 说明方向明确，但对象、字段、接口或顺序仍可调整
- `讨论中`
  - 说明存在关键待决项，暂不能作为实现依据
- `已定稿`
  - 说明接口和边界已收敛，可作为实现依据
- `已实现`
  - 说明文档对应内容已落地到代码，后续变更必须先改代码再回写文档

当前目录建议这样使用：

- `00-系统级智能化产品定义与边界.md`
- `00-全局控制平面与闭环设计.md`
- `00-智能化重构总方案.md`
- `01-05`
- `07-15`
  - 默认理解为 `草案` 或 `讨论中`
- `06-RAG写库与检索对象设计.md`
  - 优先维护为 `现状已核对`
- `16-施工附录与代码落点清单.md`
  - 默认理解为 `草案`，但应尽量保持接近可直接施工状态

## 4. 写法要求

每份文档必须显式区分以下层次：

- 当前现实
  - 只能写已经能在代码、SQL、配置中找到依据的内容
- 设计判断
  - 对现状问题的分析和取舍
- 目标设计
  - 尚未实现的新增对象、状态、接口和流程
- 分阶段落地
  - 从当前半成品到目标态的实施顺序
- 非结论项
  - 仍待验证、待选型或待压测的部分

如果文档涉及多个目标态对象，还应进一步区分：

- 当前最优收敛
  - 近期最值得优先落地的核心对象和最小闭环
- 第二阶段增强
  - 适合作为派生读模型、异步产物或产品化对象的内容
- 北极星能力
  - 远期产品方向，但不应被误读成近期硬目标

如果文档里需要描述完整目标流程，标题应尽量明确为：

- `目标态完整流程`
- `目标态完整主流程`
- `目标态完整用户旅程`

并在正文里明确说明：

- 这描述的是完整闭环，不等于第一阶段全部显式化到独立对象

如果某个对象还不存在，必须写成：

- “建议新增”
- “目标态拟引入”
- “当前未实现”

不能写成：

- “当前核心真相对象是 …”
- “系统统一收敛为 …”

除非代码或数据库里已经真实存在。

## 5. 证据约定

写“当前现实”时，至少应能对应到以下一种证据：

- Python 代码路径
- 前端代码路径
- 初始化 SQL
- Alembic 迁移
- 配置项
- 测试
- systemd 或 docker 编排文件

推荐写法如下：

- 当前查询入口在 `server/api/query/routes.py`
- 当前 REST 路由实际挂载在 `server/main.py`
- 当前持久化表来自 `docker/init-scripts/init_database_complete.sql`
- 当前部署入口以 `deploy/systemd/nl2sql-backend.service` 为准

## 6. 系统级顶层定义文档模板

系统级顶层定义文档建议按以下结构展开：

## 1. 文档信息

- 状态
- 类型
- 最后更新时间

## 2. 文档目标

- 最终产品是什么
- 为什么不能只理解成局部升级

## 3. 当前系统真实现状

- 当前已有底座
- 当前为什么仍是半成品
- 当前最关键的工程判断

## 4. 最终目标产品定义

- 产品线
- 系统边界
- 设计原则

## 5. 与其他文档的关系

- 哪些文档负责控制平面、共享契约和实施蓝图

## 7. 系统级控制平面或实施蓝图文档模板

系统级控制平面或实施蓝图文档建议按以下结构展开：

## 1. 文档信息

- 状态
- 类型
- 最后更新时间

## 2. 文档目标

- 要统一什么系统问题
- 回答哪些跨域问题

## 3. 当前现实

- 当前控制信息或交付能力分散在哪里
- 当前缺口在哪里

## 4. 目标态设计

- 全局平面、共享对象或阶段矩阵
- 当前最优收敛线
- 关键闭环
- 硬约束

## 5. 分阶段落地

- 各阶段目标
- 每阶段涉及的对象、后端、前端、测试和交付

## 6. 与其他文档的关系

- 共享契约、领域 RFC 和现状真相文档如何承接

## 8. 领域 RFC 文档模板

领域 `RFC` 文档建议按以下结构展开：

## 1. 文档信息

- 状态
- 对应总文档章节
- 最后更新时间

## 2. 文档目标

- 这个域要解决什么问题
- 为什么需要单独成域

## 3. 当前现实

- 当前实现入口
- 当前已实现能力
- 当前关键缺口

## 4. 目标设计

- 目标对象
- 状态或生命周期
- 输入输出
- 与现有代码的衔接方式
- 哪些属于第一波核心对象，哪些属于第二阶段或北极星能力

## 5. 分阶段落地

- 第一阶段做什么
- 暂不做什么
- 仍待验证什么

## 6. 风险与待决项

- 风险
- 未决问题

## 9. 共享契约与现状真相文档模板

共享契约文档建议按以下结构展开：

## 1. 文档信息

- 状态
- 类型
- 最后更新时间

## 2. 文档目标

- 哪些模块共同使用
- 当前是否已落地

## 3. 当前已实现对象

- 当前线上或仓库内已经存在的共享对象

## 4. 目标态新增对象

- 仍未实现但计划引入的对象

## 5. 标识、状态与版本规则

- 共享标识
- 状态枚举
- 幂等键
- 兼容策略

## 6. 变更规则

- 追加字段如何处理
- 破坏性变更如何处理

现状真相文档建议额外强调：

- 当前事实源分层
- 当前真实写入链路
- 当前兼容逻辑
- 当前已知限制

## 10. 命名约定

- 文档标题使用中文，可保留必要专业名词
- 文件名使用序号前缀，便于排序
- 共享对象命名优先与代码保持一致；目标态对象需在文档中标清“尚未实现”
- `json/jsonb` 容器字段统一以 `_json` 结尾
- 时间字段统一以 `_at` 结尾
- 运行对象主键命名优先显式，如 `query_id`、`release_run_id`、`evaluation_run_id`

## 11. 维护约定

- 顶层定义文档负责产品边界，不下沉到模块级字段细节
- 控制平面和实施蓝图文档负责跨域闭环与阶段顺序
- 领域 RFC 文档负责域内拆解，不重复整篇抄顶层文档
- 共享契约文档负责跨域对象与字段，不承载具体算法
- 现状真相文档负责描述当前实现，不替目标态背书
- 施工附录文档负责真实代码落点、命令、验收和回滚点，不重新发明顶层目标
- 涉及目标态对象的文档，必须同步说明“第一波核心 / 第二阶段增强 / 北极星能力”
- 代码已经实现而文档未更新时，以代码为准，并尽快修文档
- 新增文档时，先登记到 `README.md`
