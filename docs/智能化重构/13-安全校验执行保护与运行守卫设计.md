# 安全校验执行保护与运行守卫设计

## 1. 文档信息

- 状态：草案
- 对应总文档章节：`00-智能化重构总方案.md`
- 最后更新时间：2026-04-16

## 2. 文档目标

本文档回答以下问题：

- 当前系统已经有哪些执行安全和运行守卫能力
- 查询在进入数据库前后，实际有哪些保护措施
- 最终智能化产品还缺哪些安全与保护层设计
- 这些能力如何和权限、日志、状态层协同

## 3. 当前现实

### 3.1 当前已经有的执行安全面

当前代码已经不是“生成 SQL 就直接跑”，而是至少有以下保护层：

- SQL 安全校验
  - `server/security/sql_validator.py`
- 查询执行超时控制
  - `server/exec/executor.py`
- 成本守护
  - `server/exec/showplan.py`
- Dry Run 与增强校验
  - `server/enhanced_validation/loop.py`
- 快照一致性会话
  - `server/session_manager/manager.py`
- 停止与取消
  - `server/services/stop_signal_service.py`
  - `server/api/query/routes.py`

因此安全与保护能力并非空白，只是当前还比较分散。

### 3.2 SQL 安全校验的真实情况

当前 `SQLSecurityValidator` 已明确禁止：

- DML
  - `INSERT`
  - `UPDATE`
  - `DELETE`
  - `TRUNCATE`
  - `MERGE`
- DDL
  - `CREATE`
  - `ALTER`
  - `DROP`
  - `RENAME`
- DCL / 事务与危险执行
  - `GRANT`
  - `REVOKE`
  - `BEGIN`
  - `COMMIT`
  - `ROLLBACK`
  - `EXEC`
  - `CALL`

并且还会检查：

- 多语句风险
- 危险模式
  - `xp_cmdshell`
  - `OPENROWSET`
  - `BULK INSERT`
  - `SET @@`

当前这层已经构成了“只读查询保护”的基础事实。

### 3.3 当前执行超时与数据库侧保护

当前 `QueryExecutor` 会在执行前按数据库类型设置超时：

- SQL Server
  - `SET LOCK_TIMEOUT`
- PostgreSQL
  - `SET statement_timeout`
- MySQL
  - `SET SESSION max_execution_time`
- MariaDB
  - `SET SESSION max_statement_time`

这说明当前已经有数据库级运行保护，不是完全依赖应用层超时。

### 3.4 当前成本守护与用户确认

当前查询路由里已经接入成本守护：

- `server/exec/showplan.py`
- `server/api/query/routes.py`

真实行为是：

- 对高成本查询先做估算
- 超过阈值时返回 `confirm_needed`
- 用户通过 `force_execute` 再次提交后继续执行

因此“高成本确认”已经是当前稳定存在的保护面之一。

### 3.5 当前 Dry Run 与增强校验

当前已经有增强校验回路：

- 实体对齐占位器
  - `EntityLinker`
- SQL Dry Run
  - `ExecutionValidator`
- 统一入口
  - `EnhancedValidationLoop`

当前限制也要明确：

- 实体对齐仍是占位实现
- Dry Run 是执行前保护，但还没有形成完整的产品级审计对象

### 3.6 当前一致性与取消控制

当前已经有两类重要运行守卫：

- 一致性守卫
  - `DatabaseSessionManager.snapshot_scope()`
  - 用于复杂查询期间保持同一快照
- 取消守卫
  - `StopSignalService`
  - 流式查询中会持续检查停止信号

当前前端和后端的实际交互也已经支持：

- 用户停止查询
- 后端将消息状态标记为 `cancelled`
- 流式链路中断输出

### 3.7 当前导出与密钥治理的真实情况

当前系统已经涉及两类容易被忽略的安全面：

- 导出控制
  - 数据权限里已有 `can_export`
  - `frontend/src/views/Chat.vue` 支持导出 CSV
- 密钥与配置展示
  - `server/api/admin/system_config.py` 会对 API Key 做 mask
  - 模型供应商凭证在数据库中管理

但当前仍然没有形成完整设计：

- 导出审计闭环
- 导出脱敏或分级策略
- 凭证读取审计
- 集中的密钥托管策略

## 4. 当前还缺的安全与保护域

如果从最终智能化产品角度看，当前还缺以下几类守卫面：

- `policy_snapshots`
- 成本守护、Dry Run、超时、取消的统一审计对象
- 导出安全与结果二次治理
- 密钥与凭证的集中托管和读取审计
- 安全事件与故障事件的统一关联

## 5. 目标态安全与运行守卫架构

### 5.1 查询准入层

负责：

- 身份校验
- 权限校验
- 风险分级
- 是否允许进入执行链路

最终建议把以下信息固化到一次查询的准入快照里：

- 用户与组织上下文
- 命中的系统角色与数据角色
- 是否需要高成本确认
- 是否允许导出
- 是否进入降级模式

这类准入结果更适合沉淀到：

- `policy_snapshots`
- `query_sessions.state_json`

### 5.2 SQL 安全与只读执行层

负责：

- 只读校验
- 多语句风险防御
- 危险函数和系统调用拦截
- 方言级超时控制

这层不应被任何外部适配器绕过。

### 5.3 成本守护与执行前验证层

负责：

- 成本估算
- Dry Run
- 实体对齐
- 风险提示与用户确认

最终建议把以下结果统一记录：

- 预估行数
- 预估成本
- Dry Run 是否通过
- 最终是否被用户批准
- 批准动作对应的 `query_id` / 动作摘要

### 5.4 一致性、取消与资源治理层

负责：

- 快照一致性
- 取消与停止信号
- 数据库超时
- 复杂查询资源边界

最终建议所有这些结果都进入状态层和可观测性层，而不是只留在运行内存里。

还应明确：

- 运行层取消可以继续依赖停止信号
- 但产品层取消结果必须统一沉淀到 `query_sessions`
- 继续执行或重连传输时不应重建业务级查询标识

### 5.5 导出与结果安全层

负责：

- 结果是否允许导出
- 导出是否需要额外审计
- 是否需要脱敏或遮罩
- 大结果集导出边界

这层当前还没有被独立设计，但最终智能化产品一定要补。

### 5.6 密钥与凭证治理层

负责：

- 模型供应商凭证
- 外部认证提供者凭证
- 外部集成调用凭证
- 凭证读取、更新和轮换审计

当前只做到了部分配置管理，未形成完整的密钥治理体系。

## 6. 与权限和日志的关系

### 6.1 与权限文档的边界

`09-认证权限组织与多角色体系设计.md` 主要回答：

- 谁能进什么后台
- 谁能查哪些数据

本文档补的是另外一层：

- 即便已经通过权限校验，查询还能否安全执行
- 是否需要高成本确认
- 是否允许导出
- 如何取消、超时和保持一致性

### 6.2 与可观测性文档的边界

`10-可观测性日志追踪与运维工作台设计.md` 主要回答：

- 如何记录
- 如何排障

本文档回答的是：

- 记录之前，到底有哪些安全与运行守卫应该存在

## 7. 当前开发硬约束

1. 任何查询链路都不得绕过 SQL 只读校验。
2. 成本守护、Dry Run、超时、取消不应继续散落为不可回放的运行细节。
3. 导出必须继续受数据权限控制，最终还要补齐导出审计与脱敏策略。
4. 密钥展示必须默认脱敏，后续不应把真实凭证直接暴露给前端或普通日志。
5. 安全守卫和权限校验必须能通过同一个 `query_id` 或 `query_sessions` 记录关联起来。

## 8. 第一阶段建议

### 第一阶段建议实现

- 把 SHOWPLAN、Dry Run、超时、取消结果写入 `query_sessions.state_json` 或 `system_events`
- 把导出动作接入审计
- 明确“需要确认”和“允许直接执行”的风险分级
- 为模型供应商凭证和外部集成凭证补齐最小审计记录

### 第一阶段暂不建议实现

- 自建复杂安全策略引擎
- 一开始就做大而全的零信任体系
- 先上复杂 ABAC 再补基本执行守卫
