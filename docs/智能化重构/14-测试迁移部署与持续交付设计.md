# 测试迁移部署与持续交付设计

## 1. 文档信息

- 状态：草案
- 对应总文档章节：`00-智能化重构总方案.md`
- 最后更新时间：2026-04-17

## 2. 文档目标

本文档回答以下问题：

- 当前系统如何做迁移、测试和部署
- 最终智能化产品为什么必须把这些纳入总架构
- 需求、文档、数据库结构、前后端代码和运行环境应如何一起演进
- 发布、回滚和验证应如何约束

## 3. 当前现实

### 3.1 当前已经落地的迁移体系

当前仓库已经不是“只有初始化 SQL”，而是：

- Alembic 正式迁移入口
  - `alembic.ini`
  - `migrations/`
- 基线版本
  - `migrations/versions/20260414_0001_metadata_baseline.py`
- 首次建库 SQL
  - `docker/init-scripts/init_database_complete.sql`
- Docker 初始化后写入基线版本
  - `docker/init-scripts/999_stamp_alembic_baseline.sql`

这说明当前已经有“基线 SQL + Alembic”的双轨体系。

### 3.2 当前测试体系的真实情况

当前测试目录已经按三层拆分：

- `tests/unit/`
  - 编译器、LLM 客户端回退、前端静态和品牌等快速回归
- `tests/schema/`
  - 初始化 SQL、Alembic 基线和 SQL 文件回归
- `tests/integration/`
  - 真实依赖下的迁移 smoke

当前公共初始化：

- `tests/conftest.py`

当前已有的代表性测试包括：

- `tests/schema/test_init_database_sql.py`
- `tests/schema/test_migration_baseline.py`
- `tests/integration/test_migration_smoke.py`
- `tests/unit/web/test_frontend_static.py`

因此当前系统已经有“交付验证”的基本骨架。

### 3.3 当前运行部署拓扑

当前基础设施编排以 `docker/docker-compose.yml` 为准，至少包含：

- Redis
- PostgreSQL
- etcd
- MinIO
- Milvus
- Attu

当前服务托管入口包括：

- `deploy/systemd/nl2sql-infra.service`
  - 负责 `docker compose up -d`
- `deploy/systemd/nl2sql-backend.service`
  - 负责后端启动
  - 启动前执行 `alembic upgrade head`

### 3.4 当前前端交付方式

当前前端不是独立网关托管，而是：

- 前端构建产物位于 `frontend/dist/`
- 由 `server/frontend_static.py` 和 `server/main.py` 托管

因此最终交付链路必须同时覆盖：

- 前端构建
- 后端接口
- 元数据库结构
- 基础设施依赖

### 3.5 当前持续交付面的缺口

当前仓库还缺少一些正式的交付能力：

- 仓库内没有 CI 工作流目录
  - 当前未见 `.github/`
- 没有自动化发布流水线定义
- 没有统一的发布门禁清单
- 没有回滚 runbook 文档
- 没有把“文档变更 -> 迁移 -> 测试 -> 部署”串成一条正式产品交付链

### 3.6 当前还需要特别说明的一点

当前 `deploy/systemd/nl2sql-backend.service` 的 `ExecStart` 仍使用：

- `uvicorn server.main:app --host 0.0.0.0 --port 8891 --reload`

这是真实现状，但它更像开发态启动参数，不应被直接当成最终生产态架构结论。

## 4. 为什么这部分属于“最终架构”

如果没有迁移、测试、部署与回滚设计，所谓“最终智能化产品架构”就只是一张研发视角的逻辑图，无法支撑：

- 状态对象持续演进
- 权限表结构迭代
- 治理资产扩展
- Prompt / 模型配置治理发布
- 多角色后台长期演进

因此这部分不是运维附录，而是最终产品架构的正式组成部分。

## 5. 目标态交付与生命周期架构

### 5.1 变更类型分层

最终建议把变更分成至少四类：

- 结构变更
  - 元数据库 Schema、初始化 SQL、Alembic revision
- 行为变更
  - 查询链路、权限链路、治理链路、平台支撑链路
- 产品变更
  - 前端工作台、交互协议、状态对象
- 运行变更
  - Docker、systemd、环境变量、健康检查、依赖版本

每类变更都应明确其验证集和回滚方式。

### 5.2 目标态发布主链路

以下链路描述的是目标态完整发布闭环，不等于第一阶段必须同时把所有发布对象都显式化成独立表。

建议未来固定为：

1. 先更新方案与对象定义文档
2. 再落库结构和迁移
3. 再改后端逻辑
4. 再改前端交互
5. 运行 unit / schema / 必要 integration 测试
6. 对发布对象记录 `evaluation_runs` 或等价最小验证记录
7. 做前端构建和后端健康检查
8. 执行最小冒烟查询验证
9. 再进行服务发布，并记录 `release_runs` 或等价最小发布审计
10. 发布后检查 health / ready / 核心查询 / 后台关键页
11. 必要时通过 `rollback_records` 回滚

建议第一阶段把第 6 步到第 11 步继续细化为最小门禁：

- 阻断门禁
  - schema 测试
  - 与发布对象直接相关的 unit / integration / smoke
  - 权限与安全回归
- 发布记录
  - `evaluation_runs` 必须至少说明范围、结果、是否阻断
  - 发布审计必须至少说明影响对象、执行结果、回滚入口
- 发布后观察
  - health / ready
  - 核心查询 smoke
  - 后台关键工作台可访问性
  - 必要时记录观察窗口内的回滚判定

### 5.3 迁移治理策略

涉及元数据库变更时，最终要求应保持：

- Alembic revision 更新
- `docker/init-scripts/` 基线同步评估
- `tests/schema/` 回归同步更新

不能出现：

- 只改初始化 SQL，不改 Alembic
- 只改 Alembic，不补 schema 回归
- 只在数据库手工改，不回写仓库

### 5.4 测试门禁策略

最终建议把测试门禁分成三层：

- 快速门禁
  - `tests/unit/`
- 结构门禁
  - `tests/schema/`
- 运行门禁
  - `tests/integration/`
  - 健康检查
  - 最小查询 smoke

### 5.5 部署与回滚策略

最终建议明确两类回滚：

- 应用回滚
  - 回退前后端代码和服务版本
- 结构回滚
  - 回退 Alembic revision 或执行向前兼容修复

同时要明确：

- 哪些变更支持直接回滚
- 哪些变更需要前向修复
- 哪些数据变更必须先备份再发布
- 哪些回滚需要结构化记录到 `rollback_records`

建议未来再把变更分成三类处置：

- 可直接回滚
  - 代码、前端静态资源、纯读取型配置
- 需要前向修复
  - 破坏性结构变更、数据修复型迁移、已扩散到正式层的数据写入
- 必须先备份再动
  - 正式治理发布、凭证与高危配置改动、影响检索重建的数据操作

### 5.6 备份与恢复策略

当前 `docker/docker-compose.yml` 已给出 `data/` 目录备份说明。

最终仍建议把备份与恢复正式化为：

- 元数据库备份
- 向量数据备份
- 配置与 Prompt 导出
- 关键服务恢复顺序

### 5.7 当前可执行发布 runbook

在当前仓库条件下，最小可执行发布顺序建议固定为：

1. 确认本次改动涉及哪些层：
   - 文档
   - Alembic / 初始化 SQL
   - 后端接口与服务
   - 前端页面
   - systemd / docker / `.env` 关键配置
2. 先跑结构与基线命令：
   - `conda run -n nl2sql-py312 alembic -c alembic.ini upgrade head`
   - `conda run -n nl2sql-py312 pytest -q tests/schema`
3. 再跑代码级回归：
   - `conda run -n nl2sql-py312 pytest -q tests/unit`
   - 如本次改动依赖真实数据库，再跑 `conda run -n nl2sql-py312 pytest -q tests/integration/test_migration_smoke.py`
4. 再构建前端：
   - `/bin/bash -lc "cd frontend && npm run build"`
5. 本地或预发启动后端：
   - `conda run -n nl2sql-py312 uvicorn server.main:app --host 0.0.0.0 --port 8891`
6. 先验证基础探针：
   - `curl -fsS http://127.0.0.1:8891/health`
   - `curl -fsS http://127.0.0.1:8891/ready`
7. 再做产品级 smoke：
   - 打开 `/`，确认 `frontend/dist/` 已被 `server/main.py` 正常托管
   - 打开 `/admin`，确认后台入口和主要菜单可访问
   - 提交一次真实或预设的问数 smoke，确认查询日志可在 `/api/admin/query-logs` 或后台查询日志页看到
8. 若以上全部通过，再进入 systemd / 正式服务发布

第一阶段不要求这些步骤全部自动化，但要求：

- 每一步都有命令或明确人工检查项
- 每一步都能说明失败后阻断在哪
- 每一步都能说明由谁决定继续或回滚

### 5.8 当前可执行回滚 runbook

当前仓库更适合区分三类回滚：

#### 应用层回滚

适用：

- 后端逻辑问题
- 前端页面问题
- 配置读取问题

建议动作：

1. 切回上一个已验证 commit
2. 重新构建前端 `frontend/dist/`
3. 重新启动 `deploy/systemd/nl2sql-backend.service`
4. 再次检查 `/health`、`/ready`、首页和后台入口

#### 结构层回滚

适用：

- 新增迁移导致服务不可用
- 新表结构影响现有查询或后台

建议动作：

1. 先判断是否属于纯追加、可安全回退的 revision
2. 只有在确认不会破坏已写入数据时，才执行：
   - `conda run -n nl2sql-py312 alembic -c alembic.ini downgrade <previous_revision>`
3. 如果已存在真实数据写入或同步扩散，优先选择前向修复，而不是强行 downgrade

#### 治理与发布层回滚

适用：

- 候选发布后引入语义退化
- 同步后检索结果退化

建议动作：

1. 先冻结新的发布动作
2. 记录最小发布审计或 `release_runs`
3. 根据正式层写入范围执行反向变更或恢复快照
4. 必要时同步触发检索层重新构建
5. 把回滚结果沉淀到 `rollback_records` 或等价最小记录

## 6. 当前开发硬约束

1. 任何正式 Schema 变更都必须同时评估 Alembic、初始化 SQL 和 schema 测试。
2. 前后端协议变更必须同步更新文档和最小回归验证。
3. 发布不能只看进程启动成功，还要看 health、ready 和核心查询 smoke。
4. 不允许长期依赖手工数据库改动维持线上状态而不回写仓库。
5. 当前 systemd 启动参数里的开发态约定，不应直接当作最终生产态设计。

## 7. 第一阶段建议

### 第一阶段建议实现

- 把新增状态对象、策略对象和治理对象的迁移策略写入同一套交付文档
- 为关键产品链路补最小 smoke 测试清单
- 为发布增加最小验证顺序
- 明确哪些操作属于前向修复，哪些支持回滚
- 为 `evaluation_runs` 固定阻断门禁与观察窗口最小字段
- 为 `release_runs` / `rollback_records` 预留最小发布审计与回滚记录契约

### 第一阶段暂不建议实现

- 一开始就做复杂多环境发布平台
- 一次性重写全部部署方式
- 在没有最小 smoke 之前先做大规模自动化流水线
