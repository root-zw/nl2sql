# 智能化重构文档导航

## 1. 目录定位

本目录不再只服务于“智能问数主链路怎么升级”，而是服务于：

- 把当前半成品系统重构成完整的系统级智能化产品
- 让重构文档既是 `RFC`，也是后续可落地的实施蓝图入口
- 覆盖查询、治理、权限、观测、平台、安全、交付和持续进化全系统

当前必须先明确四件事：

- 代码、初始化 SQL、迁移、当前配置和当前机器状态才是现状真相源
- 当前仓库已经有不少智能能力，但它们分散在多个模块与后台中，还不是统一产品
- 本目录大部分文档仍然是目标态 `RFC`，不是“当前线上已经完整落地”
- 后续改造规模和工作量不是约束，最终目标实现才是约束

## 2. 当前总判断

基于当前代码，现阶段更准确的判断不是“已经有一个完整智能化产品”，而是：

- 当前已经有较强的智能分析执行内核
- 当前已经有治理、权限、监控、同步、模型配置和后台管理底座
- 当前系统仍然是半成品
  - 智能能力分散
  - 状态与事件分散
  - 工作台分散
  - 发布与持续进化链路分散
- 目标态不应只是“更强的 NL2SQL”
- 目标态应是“系统级智能化分析产品平台”

因此，本目录后续所有设计都以这个总目标为准：

- 智能分析执行域只是核心域之一
- 治理发布域、身份策略域、可观测与运营域、平台控制域、安全守卫域、交付进化域都属于产品主域

但这不等于：

- 所有目标态对象都要同优先级落地
- 所有产品线都要在第一阶段同时产品化
- 只要设计足够完整，就等于当前最优实施顺序已经确定

更准确的当前判断是：

- 这套文档的总体方向是对的，但不是每个解决方案都适合作为当前第一波实施目标
- 当前最优收敛线仍应围绕状态权威、动作协议、学习事实和治理入口四件事展开
- 一部分对象适合先作为派生读模型、异步产物或预留发布门禁，而不是立即上升为第一波核心表

## 2.2 2026-04-17 已实现进展

截至 2026-04-17，第一波与“统一确认”直接相关的实现已落地：

- `query_sessions`
  - 已落表、已写入主查询链路、已提供查询快照接口
- `draft_actions`
  - 已落表、已提供动作提交接口、已支持自由输入解析为有限动作
- `Chat.vue`
  - 已切到基于 `query_sessions` 的统一确认主路径
  - 已支持确认阶段自由回复
  - 已补齐 `table_resolution` / `draft_confirmation` / `execution_guard` 三个确认节点的按钮动作
  - 已支持结果态“不是这张表，重新选表”
- `draft_confirmation`
  - 已能在用户给出修改意见后自动重算草稿，并再次停在统一确认容器里等待确认
- 确认模式
  - 已新增 `CONFIRMATION_MODE`
  - 当前仓库默认值已切到 `always_confirm`
  - Chat 提问区已支持“本次确认策略”按钮组，可单次覆盖为 `adaptive` / `always_confirm`
- 自动选表纠错
  - 高置信度自动选表成功后已保留 `candidate_snapshot`
  - `change_table` 已能打断运行中查询并回退到 `table_resolution`

仍未落地的对象与能力：

- `query_drafts`
- `learning_events`
- `runtime_memories`
- `governance_candidates`
- LangGraph / 独立 Skills Runtime / 学习编排执行器

## 2.1 当前最优收敛判断

从当前代码现状出发，最值得优先落地的是：

- `query_sessions`
- `draft_actions`
- `learning_events`
- `governance_candidates`

这些对象分别解决四个当前真实缺口：

- `query_sessions`
  - 统一查询生命周期状态权威
- `draft_actions`
  - 统一用户确认、修订、继续执行的动作协议
- `learning_events`
  - 统一运行收益与反馈事实来源
- `governance_candidates`
  - 统一从运行时收益进入正式治理的入口

同时需要在第一波交互设计里明确一条重要边界：

- 会话内围绕上一份结果继续分析或做对比，属于 chat-first 的短期 `analysis_context`
- 它应先内嵌在 `query_sessions.state_json` 等状态对象中
- 它不应和长期 `runtime_memories` 混成同一层能力

更适合放在第二阶段或先以派生形态存在的内容包括：

- `query_drafts`
- `runtime_memories`
- `policy_snapshots`
- `evaluation_runs`
- `rollback_records`
- `system_events`

更适合作为北极星能力而不是近期硬目标的内容包括：

- 六条产品线全面同步产品化
- 通用 Skills Runtime
- 通用多 Agent 协作平台
- 完整的系统级 workflow 编排平台

- 再看执行域总方案和实施蓝图，明确从现状走向目标态的路线
- 再回到共享契约和施工附录，把对象、文件落点和实施动作收紧
- 如需进入查询确认交互、Prompt 与 Skill 收敛开发，优先补读 `17`
- 最后再看现状真相文档，核对正式层与检索层写入事实
- 最后进入各领域 RFC 细化设计

## 6. 当前重写优先级

为了把这套文档从“高质量 RFC”推进成“可落地实施蓝图”，建议重写优先级固定为：

1. 顶层定义层
   - 先把系统级目标、全局控制平面和实施矩阵钉住
2. 共享契约层
   - 重点是 `05-共享契约与关键对象定义.md`
3. 执行域核心文档
   - `00-智能化重构总方案.md`
   - `01-04`
4. 系统主域文档
   - `08-14`
5. 施工附录与运行手册
   - `16-施工附录与代码落点清单.md`
6. 最后再回收旧表述、统一命名、补充验收和回滚规则

## 7. 当前最需要避免的偏差

- 不要再把“智能化”默认等价于“问数链路加状态和记忆”
- 不要把权限、观测、平台、安全、交付写成外围附录
- 不要只写目标能力，不写控制平面、发布路径和回滚路径
- 不要只写局部模块设计，不写跨域共享对象和全局关联键
- 不要让前端原型、后端对象、迁移脚本和测试门禁脱节
- 不要把所有目标态对象都当成第一波必落地对象
- 不要把北极星能力误读成当前最优实施顺序

## 8. 维护规则

- 写“当前现实”时必须能在代码、SQL 或配置中找到依据
- 写“目标设计”时必须注明它是建议、阶段定案，还是尚未定案
- 顶层定义变更时，先改本文件和 `00-*` 顶层文档
- 共享对象变更时，优先改 `05-共享契约与关键对象定义.md`
- 涉及正式语义层、同步链路或检索对象变更时，必须同步检查 `06-RAG写库与检索对象设计.md`
- 进入实施阶段后，任何大项改造都应同步落到 `15-实施矩阵与阶段蓝图.md`
