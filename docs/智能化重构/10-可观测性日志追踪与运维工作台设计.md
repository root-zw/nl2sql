# 可观测性日志追踪与运维工作台设计

## 1. 文档信息

- 状态：草案
- 对应总文档章节：`00-智能化重构总方案.md`
- 最后更新时间：2026-04-17

## 2. 文档目标

本文档回答以下问题：

- 当前系统哪些日志和运行记录已经存在
- 一次查询的过程当前如何被记录
- 后台如何查看日志和统计
- 最终智能化产品的可观测性应如何分层

## 3. 当前现实

### 3.1 应用日志

当前应用日志使用：

- `structlog`
- 输出到标准输出
- FastAPI / systemd 部署时可被 `journalctl` 或容器日志采集

初始化入口：

- `server/main.py`

### 3.2 查询 trace

当前已经有查询追踪器：

- `server/trace/tracer.py`
- `server/trace/api.py`

当前特点：

- 追踪步骤保存在内存
- 追踪文件落到项目根目录 `trace_logs/`
- 支持 `original_query_id` 续接
- 支持导出 markdown 格式

当前接口：

- `GET /api/debug/traces`
- `GET /api/debug/traces/{query_id}`
- `GET /api/debug/traces/{query_id}/steps/{step_index}`

### 3.3 查询日志

当前已经有持久化查询日志：

- `query_history`

写入入口：

- `server/api/query/history.py`
- `server/api/query/routes.py`

后台查看入口：

- `server/api/admin/monitor.py`
- `frontend/src/views/admin/components/QueryLogs.vue`

当前支持：

- 分页筛选
- 按用户、连接、时间搜索
- 查看原问题、SQL、耗时、状态
- 查看结果预览
- 导出 CSV

### 3.4 会话与运行态记录

当前还存在两类运行态记录：

- `conversation_messages`
  - 用户可见消息和结果落地
- `active_queries`
  - 正在运行、取消中、已完成等执行状态

相关入口：

- `server/api/conversations.py`
- `server/services/conversation_service.py`

### 3.5 配置与同步审计

当前还存在：

- `metadata_change_log`
  - 配置变更历史
- `milvus_pending_changes`
  - 待同步记录
- `milvus_sync_history`
  - 同步执行历史
- `sync_errors`
  - 同步错误

### 3.6 系统健康与统计

当前后台监控接口：

- `GET /api/admin/monitor/stats`
- `GET /api/admin/monitor/health`
- `GET /api/admin/monitor/query-logs`

相关实现：

- `server/api/admin/monitor.py`
- `server/monitoring/metrics_collector.py`

## 4. 当前记录链路如何理解

如果按一次查询来看，当前至少会留下以下记录：

1. WebSocket 或 REST 请求日志
2. `QueryTracer` 步骤记录
3. `conversation_messages` 占位消息和结果消息
4. `active_queries` 运行态记录
5. `query_history` 持久化查询日志

如果涉及治理，还会留下：

6. `metadata_change_log`
7. `milvus_pending_changes`
8. `milvus_sync_history`

因此当前系统不是没有日志，而是记录分散在多套对象里。

## 5. 最终产品推荐的可观测性分层

### 5.1 用户可见层

给最终用户看的记录应包括：

- 会话列表
- 消息时间线
- 当前查询状态
- 当前确认动作
- 结果说明

这层不应该暴露底层 trace 和技术细节。

### 5.2 产品状态层

给产品和排障使用的记录应包括：

- `query_sessions`
- `query_drafts`
- `draft_actions`
- `learning_events`

这层回答的是“用户和系统在产品语义上发生了什么”。

当前第一阶段更适合先稳定：

- `query_sessions`
- `draft_actions`
- `learning_events`

`query_drafts` 更适合作为第二阶段显式对象，第一阶段也可先以读模型或界面视图存在。

这层还应明确一条约束：

- 同一条产品级查询在多次确认、修订和继续执行过程中应保持同一个 `query_id`
- 用户可见 assistant 消息应尽量保持同一个 `message_id`

### 5.3 技术 trace 层

给研发与排障使用的记录应包括：

- 检索步骤
- 解析步骤
- 权限检查步骤
- 编译步骤
- 执行步骤
- 解释与叙述步骤

这层回答的是“技术管线具体怎么跑的”。

### 5.4 运营统计层

给运营和管理看的记录应包括：

- 查询成功率
- 查询耗时
- 确认率
- 取消率
- 缓存命中率
- 发布收益
- 候选审核吞吐

### 5.5 审计层

给审计和安全看的记录应包括：

- 配置变更
- `policy_snapshots`
- `release_runs`
- `evaluation_runs`
- `rollback_records`
- `system_events`

## 6. 目标态需要补齐的记录对象

当前最需要补齐的不是“更多普通日志”，而是产品级记录对象：

- `query_sessions`
  - 生命周期主状态
- `query_drafts`
  - 当前系统理解对象
- `draft_actions`
  - 用户操作与系统动作
- `learning_events`
  - 反馈、偏好、修订、执行结果
- `governance_candidates`
  - 候选生命周期
- `policy_snapshots`
  - 本次查询、发布或高危动作应用了哪些角色与过滤
- `release_runs`
  - 发布运行对象
- `evaluation_runs`
  - 发布前后验证对象
- `rollback_records`
  - 回滚运行对象
- `system_events`
  - 跨域发布、回滚、降级、恢复事件

其中当前更适合第一波优先补齐的是：

- `query_sessions`
- `draft_actions`
- `learning_events`
- `governance_candidates`

更适合作为第二阶段预留或产品化对象的是：

- `query_drafts`
- `policy_snapshots`
- `release_runs`
- `evaluation_runs`
- `rollback_records`
- `system_events`

## 7. 目标态工作台设计

### 7.1 查询日志工作台

保留并增强当前 `QueryLogs`：

- 原问题
- SQL
- 执行结果
- 查询状态
- 结果预览
- 关联 trace
- 关联会话
- 关联 `policy_snapshots`
- 如果后续接入动作协议，还应能关联最近一次 `draft_actions`

### 7.2 Trace 工作台

当前已有 trace API，但缺少产品化 UI。

目标态建议补：

- 按 `query_id` 搜索
- 步骤时间线
- 输入输出对照
- markdown 视图
- 错误步骤高亮
- 关联用户与连接

### 7.3 治理与同步工作台

目标态建议统一查看：

- 最近 `release_runs`
- 最近 `evaluation_runs`
- 待同步数量
- 同步失败队列
- 同步重试
- 变更来源
- 回滚入口与 `rollback_records`
- 发布门禁是否通过
- 观察窗口中的回滚信号

### 7.4 权限审计工作台

目标态建议查看：

- 某次查询到底命中了哪些数据角色
- 注入了哪些行级过滤
- 拦截了哪些字段
- 是否发生权限冲突
- 关联的 `policy_snapshots`

## 8. 排障路径建议

如果一条查询有问题，建议固定按以下顺序排查：

1. 查询日志工作台看 `query_history`
2. 看会话消息是否成功落地
3. 看 trace 是否在 `trace_logs/` 或 debug API 中存在
4. 看权限是否注入或冲突
5. 看执行 SQL 和结果预览
6. 若问题与语义资产有关，再看配置变更和同步历史

## 9. 当前开发硬约束

1. 新增状态层时必须同步设计观测对象。
2. 用户可见时间线、产品状态时间线、技术 trace 不能混为一层。
3. 所有高价值动作都必须可通过 `query_id` 或后续 `query_sessions` 记录串起来。
4. 管理后台的“查询日志”不能停留在最终 SQL 层，最终应能关联 Draft、权限、候选和发布。

## 10. 第一阶段建议

### 第一阶段建议实现

- Trace 工作台原型
- 查询日志到 trace 的跳转
- 权限决策摘要落库
- 发布与同步链路可视化
- `release_runs` / `evaluation_runs` / `rollback_records` / `system_events` 契约预留
- 统一 `query_id` 关联查询日志、trace、权限和后续动作

### 第一阶段暂不建议实现

- 自建复杂观测平台
- 全量事件流平台重构
