# 系统级智能化产品定义与边界

## 1. 文档信息

- 状态：草案
- 类型：系统级顶层定义文档
- 最后更新时间：2026-04-17

## 2. 文档目标

本文档回答以下问题：

- 最终要建设的到底是什么产品
- 为什么目标不能只停留在“智能问数主链路升级”
- 整个系统哪些域都必须被纳入智能化产品范畴
- 哪些内容不属于本轮目标

### 3.2 当前系统为什么仍然是半成品

虽然底座很多，但它们目前仍然没有被统一成一个完整的系统级智能化产品：

- 查询域有智能内核，但缺统一状态和动作平面
- 后台已有多个工作台，但还不是面向角色与闭环的产品化工作面
- 治理与同步已存在，但运行收益还不能稳定进入正式治理闭环
- 权限与组织已存在，但还没有成为全系统统一策略平面
- 日志、trace、监控已存在，但还没有成为统一的诊断与运营产品面
- 配置、缓存、词典、外部适配已存在，但还没有形成平台控制平面
- 测试、迁移、部署已存在，但还没有形成持续进化产品链路

### 3.3 当前最关键的工程结论

当前系统的问题不是“能力太少”，而是：

- 全局控制平面缺失
- 跨域对象缺失
- 工作台边界缺失
- 发布、评测、回滚闭环缺失

因此最终目标不能写成：

- “把问数功能再做强一点”

而应写成：

- “把整个系统重构成可治理、可演进、可发布、可诊断的系统级智能化分析产品”

## 4. 最终目标产品定义

### 4.1 最终产品不只是 NL2SQL 服务

最终产品应同时是以下几条产品线的组合体：

1. 智能分析执行产品线
2. 智能治理与发布产品线
3. 智能身份、组织与策略产品线
4. 智能观测、诊断与运营产品线
5. 智能平台控制与外部接入产品线
6. 智能安全守卫与持续进化产品线

### 4.2 “整个系统都要智能化”具体意味着什么

不是只有查询链路需要智能化，而是以下各域都要具备结构化、可学习、可发布、可审计的能力：

- 执行域
  - 能理解、确认、修订、执行、解释和持续学习
- 治理域
  - 能把运行收益转成候选、审核、发布、回滚和评估
- 权限域
  - 能把身份、组织、角色、属性和策略固化成统一决策快照
- 观测域
  - 能从产品语义、技术 trace 和运营指标三层复盘系统行为
- 平台域
  - 能统一控制配置、缓存、词典、事件、适配器和能力开关
- 安全域
  - 能统一管理准入、只读校验、成本守护、导出、凭证和取消控制
- 交付域
  - 能把迁移、测试、发布、评测、回滚纳入同一条产品进化链

### 4.3 最终产品的核心特征

最终产品至少应满足：

- 每个关键域都不是黑盒模块，而是明确的产品工作面
- 每次查询、发布、策略变更、回滚都可被状态对象和审计对象关联
- 运行时收益不会长期停留在补丁层，而会进入治理层
- 配置、策略、模型、Prompt、检索对象都属于正式治理资产
- 测试、评测、发布、回滚不是附属流程，而是产品主流程的一部分

### 4.4 最终产品定义不等于当前最优实施顺序

本文件定义的是最终目标边界，不是“所有能力都应立即并行建设”的实施结论。

基于当前代码现状，更合理的当前最优收敛应是：

- 先把执行域的权威状态对象收敛到 `query_sessions`
- 再把确认、修订、继续执行收敛到 `draft_actions`
- 再把运行事实收敛到 `learning_events`
- 最后把可发布收益收敛到 `governance_candidates`

以下内容更适合作为第二阶段增强，而不是第一波核心实施目标：

- `query_drafts`
- `runtime_memories`
- `policy_snapshots`
- `evaluation_runs`
- `rollback_records`
- `system_events`

以下内容更适合作为北极星能力，而不是当前硬目标：

- 六条产品线同步全面产品化
- 通用 Skills Runtime
- 通用多 Agent 协作平台
- 脱离当前主链路事实的大一统工作流框架化编排

## 5. 系统边界

### 5.1 本轮目标必须覆盖的边界

本轮系统级智能化产品蓝图必须覆盖：

- 用户前台
- 智能分析执行链路
- 管理后台与角色工作台
- 正式治理资产与发布闭环
- 身份、组织、权限与策略
- 观测、trace、日志、审计、监控
- 平台配置、缓存、词法、事件、外部适配
- 安全守卫、取消、导出、凭证治理
- 迁移、测试、部署、发布、回滚、评测

### 5.2 当前不应纳入硬目标的项

以下内容当前不应被写成必须实现的硬目标：

- 自由自治式通用 Agent 操作系统
- 无边界的多 Agent 自发协作平台
- 全自动无需审核的正式治理发布
- 先于控制平面的复杂框架化编排

这些都可以是远期候选，但不是当前顶层定案。

## 6. 目标态设计原则

### 6.1 先系统化，再局部强化

最终目标优先，但实施上必须坚持：

- 先统一全局控制平面
- 再收敛共享对象与工作台边界
- 再补每个域的智能闭环

这也意味着：

- “覆盖所有域”不等于“所有域同优先级同时建设”
- 第一波最优动作应优先解决状态权威、动作协议、学习事实和治理入口
- 其他域在第一阶段更多是跟随共享契约补齐接口、工作台入口和门禁，而不是先全面产品化

### 6.2 正式层与运行时层分层

必须坚持：

- 运行时记忆不等于正式语义
- 在线收益不直接越权写正式层
- 正式层变更必须进入发布、同步、评测和回滚链路

### 6.3 人机协作是产品内建能力

最终产品里：

- 确认
- 修订
- 审核
- 发布
- 回滚

都不应是旁路操作，而应是产品主流程中的结构化动作。

### 6.4 交付能力是产品能力

迁移、测试、发布、评测、回滚不应被视为运维附录，而应被定义为：

- 智能化产品持续进化的正式能力
