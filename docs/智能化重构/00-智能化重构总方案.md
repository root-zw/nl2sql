# 智能化重构总方案

## 1. 文档信息

- 状态：草案
- 类型：系统级智能化产品蓝图下的执行域总方案
- 最后更新时间：2026-04-17

## 2. 文档定位

本文档回答两个问题：

- 当前智能分析执行域真实已经做到什么程度
- 在整个系统级智能化产品蓝图下，执行域建议按什么顺序推进

本文档不再承担以下职责：

- 不再独自承担整个系统的最高层定义
- 不把未来设计写成当前实现
- 不把讨论中的框架选型写成最终定案
- 不宣称已经找到“最优解”

当前更准确的说法是：

- 项目已经有较强的智能分析执行内核
- 项目仍缺少执行域的产品级状态层、统一动作层和可持续学习闭环
- 整个系统的更高层定义应以上层文档为准
- 执行域重构方向已经比较清晰，但具体实现路径仍需分阶段验证

当前本目录更高一层的顶层文档是：

## 3. 当前系统真实现状

### 3.1 在线查询主链路

当前核心链路的真实入口和关键模块如下：

- 后端入口：`server/main.py`
- 查询接口：`server/api/query/routes.py`
- 表选择：`server/api/query/table_selection.py`
- NL2IR：`server/nl2ir/parser.py`
- 复杂度路由：`server/query_complexity/router.py`
- SQL 编译：`server/compiler/compiler.py`
- 成本守护：`server/exec/showplan.py`
- SQL 执行：`server/exec/executor.py`
- 会话与消息：`server/services/conversation_service.py`
- 查询追踪：`server/trace/tracer.py`
- 前端聊天页：`frontend/src/views/Chat.vue`

当前 REST 查询路由的实际挂载方式是：

- 路由函数：`POST /query`
- 应用挂载前缀：`/api`
- 对外路径：`POST /api/query`

### 3.2 当前线上协议

按当前代码，主查询协议仍然是：

- 输入：`QueryRequest`
- 输出：`QueryResponse`

当前 `QueryResponse.status` 的真实枚举为：

- `success`
- `confirm_needed`
- `table_selection_needed`
- `error`

当前真正在线工作的两类确认主要是：

- 表选择确认
- 高成本执行确认

当前不存在稳定落地的语义级 `Query Draft` 主链路。

### 3.3 当前持久化对象

当前与查询链路直接相关的持久化对象主要是：

- `conversations`
- `conversation_messages`
- `active_queries`
- `query_history`
- `qa_few_shot_samples`
- `qa_few_shot_feedback`

当前与治理和同步相关的正式对象主要是：

- `fields`
- `field_enum_values`
- `table_relationships`
- `global_rules`
- `metadata_change_log`
- `milvus_pending_changes`
- `milvus_sync_history`

当前尚未实现以下目标态对象：

- `query_sessions`
- `query_drafts`
- `draft_actions`
- `learning_events`
- `runtime_memories`
- `governance_candidates`

### 3.4 当前已经具备的能力

基于当前代码，系统已经具备以下关键能力：

- 多数据库连接和动态语义模型装载
- LLM 表选择与向量表选择双链路
- 层次化检索、枚举召回和 few-shot 检索
- NL2IR 解析、复杂度路由、直接 SQL 与复杂拆分执行
- 成本守护与执行确认
- 会话消息占位、查询取消、运行中状态管理
- 查询追踪、结果叙述、历史归档
- 模型供应商管理、Prompt 模板管理
- 元数据导入导出、Milvus 全量和增量同步

### 3.5 当前关键缺口

当前真正缺的不是“再多接几个模型能力”，而是以下四类产品级能力：

- 显式的查询生命周期状态
- 用户动作驱动的继续执行机制
- 面向下一次问答的统一学习事实层
- 从运行时收益进入正式治理的闭环

如果从完整产品角度看，当前还缺少另外四类外围层：

- 面向数据管理员和审核者的治理工作台与发布流
- 面向不同角色的统一权限决策与审计闭环
- 面向研发、运营和排障的可观测性与 trace 工作台
- 面向最终用户和后台角色的完整产品交互与操作规范

如果从完整系统角度继续外扩，当前还缺少另外三类系统级外围层：

- 面向配置、缓存、词典、实时事件和外部适配的统一平台支撑层
- 面向 SQL 安全、成本守护、Dry Run、一致性与取消控制的执行保护层
- 面向测试、迁移、部署、回滚和发布验证的持续交付层

## 4. 当前最关键的工程判断

### 4.1 第一优先级不是框架

当前最大的工程问题不是框架名称，而是：

- 查询大总控过重
- 状态分散在消息、运行表、trace 和请求上下文里
- 用户动作的协议没有统一
- 结果后学习只有局部样本沉淀，没有统一事件层

因此第一优先级应是：

1. 把现有查询主链路拆成明确节点
2. 明确节点输入输出
3. 把状态对象从“隐含在过程里”变成“显式对象”

### 4.2 当前不该做的事

当前阶段不建议做以下动作：

- 把系统整体重写成自由 Agent
- 用 Prompt 继续吸收所有复杂性
- 先上多 Agent 再补状态层
- 让在线能力直接改写正式语义层

### 4.3 当前应该保留的部分

当前应尽量保留的部分是：

- retrieval
- parser
- query router
- compiler
- executor
- permission
- cost guard
- Prompt / 模型供应商治理
- 元数据正式层与现有同步链路

一句话：应优先做“外层重构，内核保留”。

## 5. 目标态建议

### 5.1 目标不是“更强一点的 NL2SQL”

更合理的目标是把当前系统升级成：

- 有显式状态
- 支持确认和修订
- 能记录结构化动作
- 能沉淀学习事实
- 能把稳定收益回写到正式治理层

### 5.2 目标态建议新增的对象

基于当前代码，目标态建议新增以下对象：

- `query_sessions`
  - 查询生命周期主状态
- `query_drafts`
  - 用户可见的当前理解草稿
- `draft_actions`
  - 用户和系统对草稿或执行的结构化动作
- `learning_events`
  - 统一学习事实层
- `runtime_memories`
  - 面向下一次问答的运行时记忆层
- `governance_candidates`
  - 治理待审层

这些对象当前都还未实现，因此这里只能作为建议设计，不应视为现状。

### 5.3 当前最优收敛的核心对象

如果从“当前代码最缺什么”而不是“目标态最终要有什么”来排序，优先级应收敛为：

1. `query_sessions`
2. `draft_actions`
3. `learning_events`
4. `governance_candidates`

原因是：

- `query_sessions`
  - 先解决状态权威分散在 `conversation_messages`、`active_queries`、`tracer` 和请求过程里的问题
- `draft_actions`
  - 先解决确认、修订、继续执行没有统一协议的问题
- `learning_events`
  - 先解决收益事实只能散落在历史、反馈和 few-shot 样本里的问题
- `governance_candidates`
  - 先解决运行收益无法稳定进入正式治理入口的问题

以下对象更适合作为第二阶段增强，而不是第一波核心对象：

- `query_drafts`
  - 可先由 `query_sessions.state_json` 和 `draft_actions` 派生读模型承接
- `runtime_memories`
  - 可先作为异步提炼结果，而不是和状态层、正式语义层并列成第三套权威
- `evaluation_runs`
  - 先保留最小阻断语义，后续再产品化
- `policy_snapshots`
  - 在查询、发布、高危动作统一接入前，先避免过早扩表
- `release_runs`
  - 可以先由治理候选发布流程承接最小发布记录
- `rollback_records`
  - 可以在发布产品化阶段再显式化
- `system_events`
  - 可以先由现有日志、trace 和关键对象关联承接

### 5.4 目标态建议的模块边界

建议未来逐步形成以下边界：

- 编排层
  - 负责查询生命周期、状态推进、动作消费
- 执行内核
  - 继续负责检索、解析、编译、执行
- 学习层
  - 负责事件记录与运行时收益提炼
- 治理层
  - 负责候选审核、正式发布和回写现有正式层
- 身份与权限层
  - 负责认证、组织、系统角色、数据角色和查询权限裁剪
- 可观测性层
  - 负责日志、trace、查询日志、审计、指标与排障入口
- 工作台层
  - 负责用户问数界面、治理后台、权限后台、模型配置后台和运维后台
- 平台支撑层
  - 负责系统配置、缓存、分词器、WebSocket 事件、健康检查和外部适配
- 执行保护层
  - 负责 SQL 安全、成本守护、Dry Run、快照一致性、取消控制和导出保护
- 持续交付层
  - 负责迁移、测试、部署、发布验证、回滚和运行基线

## 6. 框架与选型结论

### 6.1 当前没有“最终已验证框架结论”

基于当前仓库状态，以下结论更准确：

- 当前还没有任何编排框架落地
- 当前依赖里也没有 `LangGraph`、`LlamaIndex Workflows` 或 Agent SDK
- 因此框架选型仍是候选项，不是实现事实

### 6.2 当前可接受的表述

目前可以保留的判断只有：

- 如果后续确实需要图式状态编排，`LangGraph` 是可优先评估的候选项
- 但只有在内部节点边界已经抽清之后，才值得接入

### 6.3 当前不应写成定论的表述

当前不应再写成文档定论的内容包括：

- “主编排骨架已经确定为 LangGraph”
- “当前最优解就是 LangGraph + Skill Protocol”
- “多 Agent 不是最优”

这些都属于待验证设计判断，而不是当前仓库已经证明的结论。

## 7. RAG 与正式语义层的当前判断

这部分当前判断相对更稳定：

- PostgreSQL 仍是正式事实源
- Milvus 仍是可重建的检索服务层
- 在线链路不应直接写 Milvus
- 正式治理发布应优先回写现有正式语义层，再由同步链路扩散到 Milvus

这一判断由以下现实支撑：

- `qa_few_shot_samples` 先落 PostgreSQL
- 触发器写入 `milvus_pending_changes`
- 同步服务统一构造实体后写 Milvus

因此当前最贴近实现的底座文档仍是 `06-RAG写库与检索对象设计.md`。

## 8. 建议的阶段路线

### 阶段 1：拆出显式编排边界

目标：

- 不引入新框架
- 先把现有查询大总控拆成更明确的内部节点

建议动作：

- 抽离接入、表选择、解析、路由、编译守护、执行、结果记录几个阶段
- 统一这些阶段的输入输出对象

### 阶段 2：引入最小状态对象

目标：

- 先解决“状态到底在哪里”

建议动作：

- 新增最小 `query_sessions`
- 让取消、恢复、继续执行先有统一状态来源
- 同时明确 `query_sessions`、`conversation_messages`、`active_queries`、`QueryTracer` 的权威边界

### 阶段 3：引入最小 Draft 读模型与动作协议

目标：

- 把“当前理解”变成可见对象

建议动作：

- 新增 `draft_actions`
- 先统一表选择、执行批准和草稿确认三类动作
- 先由 `query_sessions.state_json` 加 `draft_actions` 派生最小可读草稿
- 如果现有读模型无法支撑前端交互，再补最小 `query_drafts`
- 固定动作写入、状态推进和异步派生的最小事务边界

### 阶段 4：引入学习事实层

目标：

- 不再只靠 `query_history` 和 few-shot 样本表达学习

建议动作：

- 新增 `learning_events`
- 先记录查询提交、草稿生成、动作应用、执行完成、结果反馈
- 先固定 `learning_events -> governance_candidates` 的主顺序
- 如需运行时增强，再在中间插入最小 `runtime_memories`

### 阶段 5：引入运行时记忆与治理候选

目标：

- 把即时收益和正式治理分层

建议动作：

- 新增 `governance_candidates`
- 视收益提炼复杂度决定是否显式化最小 `runtime_memories`
- 发布仍回写现有正式层
- 为 `evaluation_runs` 和回滚阈值预留阻断门禁语义

### 阶段 6：再评估是否需要接入编排框架

只有在以下前提成立时再评估：

- 状态对象已经稳定
- 节点边界已经清晰
- 协议已经独立于框架

## 9. 当前最小可执行建议

如果只保留一条最务实的路线，建议是：

1. 不推翻现有执行内核
2. 先拆 `server/api/query/routes.py`
3. 先建立最小状态层 `query_sessions`
4. 再建立最小动作层 `draft_actions`
5. 然后补学习事实层 `learning_events`
6. 再补治理入口 `governance_candidates`
7. 最后才考虑 `query_drafts`、`runtime_memories`、框架化和技能化

## 10. 验收标准

这套重构方案要成立，至少要满足以下结果：

- 查询状态不再只散落在消息、运行表和 tracer 中
- 用户动作可以基于持久状态推进主链路
- 学习事实不再只靠 few-shot 样本和 query history 旁路表达
- 治理收益能回写到现有正式语义层，而不是停留在运行时补丁
- 配置变化、缓存失效和生效时机能够被说明和追踪

## 11. 非结论项

截至当前版本，以下内容仍然不是定论：

- 是否一定要上 `LangGraph`
- 是否需要独立 `skills/` 目录
- `query_sessions` 是否应数据库单写还是数据库加缓存双写
- `runtime_memories` 自动应用阈值
- 治理候选的审批工作台形态

这些都应在后续实现和评审过程中继续验证。
