# 治理域与发布链路设计

## 1. 文档信息

- 状态：草案
- 对应总文档章节：`00-智能化重构总方案.md`
- 最后更新时间：2026-04-17

## 2. 文档目标

本文档回答以下问题：

- 当前系统已经有哪些治理资产
- 最终智能化产品到底需要治理哪些内容
- 治理结果如何发布到正式层、检索层、模型层和在线链路
- 后台工作台应该如何拆分

## 3. 当前现实

### 3.1 当前已经存在的正式治理资产

当前仓库已经有一套正式治理底座，不是空白：

- 数据源与域
  - `database_connections`
  - `business_domains`
- 表与字段
  - `db_tables`
  - `db_columns`
  - `fields`
- 枚举与关系
  - `field_enum_values`
  - `table_relationships`
- 规则
  - `global_rules`
- 模型与 Prompt
  - `model_providers`
  - `provider_credentials`
  - `provider_models`
  - `scenario_model_configs`
  - `prompt_templates`
- 审计与同步
  - `metadata_change_log`
  - `milvus_pending_changes`
  - `milvus_sync_history`

### 3.2 当前已经有的治理入口

当前后台已经有以下入口：

- 数据源管理
  - `server/api/admin/datasources.py`
- 统一元数据树
  - `server/api/admin/unified_metadata.py`
- 字段与枚举管理
  - `server/api/admin/fields.py`
- 关系自动识别与确认
  - `server/api/admin/joins.py`
- 规则管理与校验
  - `server/api/admin/rules.py`
- Excel 导入导出
  - `server/api/admin/metadata_io.py`
- 模型与 Prompt 管理
  - `server/api/admin/model_providers.py`
  - `server/api/admin/prompts.py`

### 3.3 当前缺失的部分

当前系统仍然没有：

- 运行时候选池
- 审核工作流
- 发布审批工作台
- 候选到正式层的统一发布适配器
- `release_runs`
- `evaluation_runs`
- `rollback_records`
- 发布收益评估与回滚工作台

## 4. 最终产品需要治理的对象

### 4.1 正式语义资产

这类资产一旦发布，就应成为正式真相的一部分：

- 业务域归属
- 表的显示名、描述、标签、年度信息
- 字段语义类型、显示名、同义词、默认聚合、展示优先级
- 枚举标准值、显示值、同义词、包含值映射
- 表关系、Join 类型、关系名称、人工确认状态
- 默认过滤规则
- 派生指标规则
- 自定义业务指令

### 4.2 检索资产

这类资产不一定直接面向业务用户，但会影响检索质量：

- Milvus 语义实体
- 枚举值向量实体
- few-shot 样本实体
- 重建策略、增量同步状态和失败重试信息

### 4.3 运行时候选资产

这类资产来自在线收益，但还不能直接视为正式标准：

- 高频字段别名建议
- 高频枚举归一化建议
- 默认过滤建议
- Join 候选建议
- 派生指标候选建议
- few-shot 样本候选
- Prompt / 模型场景配置建议

### 4.4 模型与 Prompt 资产

最终智能化产品不能只治理语义元数据，还要治理模型配置：

- 场景模型选择
- 模型供应商与凭证
- 场景级 Prompt 模板
- Prompt 版本历史
- 环境变量兜底配置

## 5. 治理后如何作用于大模型和在线链路

### 5.1 作用于检索层

正式元数据发布后，变化链路应是：

1. 写正式表
2. 记录 `metadata_change_log`
3. 触发 `milvus_pending_changes`
4. 同步服务构造实体
5. 写入 Milvus
6. 检索器在后续查询中消费新的检索对象

当前相关代码依据：

- `server/sync/`
- `server/api/admin/milvus.py`
- `server/sync/unified_sync_service.py`

### 5.2 作用于解析与编译层

正式规则发布后，应通过统一加载器进入在线链路：

- `global_rules`
  - 当前仓库里的正式规则主入口
- 代码中约定的可选 YAML 兜底文件
  - `server/utils/global_rules_loader.py` 支持从约定位置加载
  - 但当前仓库未提供该文件，因此当前工作树下实际仍以 `global_rules` 表为主
- 消费位置
  - `server/nl2ir/parser.py`
  - `server/nl2ir/hierarchical_retriever.py`
  - `server/nl2ir/vector_table_selector.py`
  - `server/compiler/compiler.py`
  - `server/compiler/rules.py`
  - `server/formatter/result_formatter.py`

因此治理发布不是“写一条配置就结束”，而是必须明确它作用到哪些在线节点。

### 5.3 作用于 Prompt 层

Prompt 不应再长期承载隐式业务补丁。

合理分工应是：

- 稳定业务语义
  - 回写 `fields`、`field_enum_values`、`table_relationships`、`global_rules`
- 场景模板
  - 回写 `prompt_templates`
- 在线调用
  - 由 `server/utils/prompt_loader.py` 按“数据库激活版优先，文件版兜底”的顺序加载

### 5.4 作用于模型层

模型治理的正式入口是：

- `model_providers`
- `provider_credentials`
- `provider_models`
- `scenario_model_configs`

在线调用应只读取正式配置，不直接从运行时请求里决定模型。

## 6. 目标态治理闭环

### 6.1 候选层

建议新增：

- `governance_candidates`

建议最小字段：

- `candidate_id`
- `candidate_type`
- `target_object_type`
- `target_object_id`
- `scope_type`
- `scope_id`
- `suggested_change_json`
- `evidence_summary`
- `evidence_payload_json`
- `support_count`
- `confidence_score`
- `status`
- `created_at`
- `reviewed_at`
- `reviewed_by`

### 6.2 审核层

建议至少有以下审核动作：

- `approve`
- `reject`
- `request_more_evidence`
- `publish_now`
- `publish_later`
- `rollback`

### 6.3 发布运行层

建议新增：

- `release_runs`

建议最小字段：

- `release_run_id`
- `release_type`
- `source_type`
- `source_ids_json`
- `policy_snapshot_id`
- `status`
- `plan_json`
- `result_json`
- `triggered_by`
- `created_at`
- `started_at`
- `completed_at`

### 6.4 评测层

建议新增：

- `evaluation_runs`

建议最小字段：

- `evaluation_run_id`
- `evaluation_type`
- `target_object_type`
- `target_object_id`
- `release_run_id`
- `status`
- `input_scope_json`
- `summary_json`
- `created_at`
- `started_at`
- `completed_at`

建议第一阶段至少固定以下 `evaluation_type`：

- `schema_guard`
- `mapping_regression`
- `query_smoke`
- `permission_regression`
- `online_observation`

并建议在 `summary_json` 中至少统一：

- 评测样本范围
- 通过 / 失败数量
- 关键指标摘要
- 是否阻断发布
- 建议动作

### 6.5 回滚层

建议新增：

- `rollback_records`

建议最小字段：

- `rollback_id`
- `release_run_id`
- `target_object_type`
- `target_object_id`
- `policy_snapshot_id`
- `reason`
- `snapshot_json`
- `status`
- `triggered_by`
- `created_at`
- `completed_at`

### 6.6 发布适配器层

发布不应直接“生成一条抽象知识”，而应翻译为正式表变更：

- 枚举归一化候选
  - 回写 `field_enum_values`
- 字段别名候选
  - 回写 `fields.synonyms`
- Join 候选
  - 回写 `table_relationships`
- 默认过滤候选
  - 回写 `global_rules`
- 派生指标候选
  - 回写 `global_rules`
- Prompt 场景建议
  - 回写 `prompt_templates`
- 模型场景建议
  - 回写 `scenario_model_configs`

### 6.7 同步与扩散层

发布完成后的扩散顺序建议固定：

1. 发布前验证并记录 `evaluation_runs`
2. 正式表提交并记录 `release_runs`
3. 记录 `metadata_change_log`
4. 触发 `milvus_pending_changes`
5. 同步服务重建实体
6. 查询侧加载新资产
7. 必要时通过 `rollback_records` 回滚

### 6.8 发布门禁与回滚触发建议

为了避免 `evaluation_runs` 变成“有记录但不挡发布”的容器，建议固定：

- `schema_guard`
  - 不通过则阻断发布
- `mapping_regression`
  - 不通过则阻断发布
- `permission_regression`
  - 不通过则阻断发布
- `query_smoke`
  - 不通过则阻断发布
- `online_observation`
  - 不一定阻断首发，但必须带预先声明的观察窗口与回滚阈值

建议把阈值声明放进发布计划或评测摘要，而不是上线后凭感觉解释。

### 6.9 第一阶段可接受的收益判定方式

第一阶段不需要一开始就做复杂评测平台，但至少要能回答：

- 这次发布影响哪些对象
- 发布前验证是否全部通过
- 发布后观察哪些指标
- 哪些退化会触发人工复核
- 哪些严重退化会直接触发回滚

## 7. 后台工作台拆分建议

### 7.1 当前已有工作台

- 元数据管理
  - 统一元数据树、字段编辑、枚举编辑、关系确认、规则管理
- 模型配置
  - Prompt 管理、模型供应商、环境变量视图
- Milvus 同步
  - 健康、统计、同步控制
- 配置历史
  - `metadata_change_log`

### 7.2 目标态建议新增工作台

- 候选池
  - 按类型、连接、业务域、证据强度过滤
- 审核台
  - 对比“当前正式值”和“候选建议值”
- 发布台
  - 记录每次 `release_runs`、影响范围、同步状态、回滚入口
- 收益评估台
  - 基于 `evaluation_runs` 对比发布前后命中率、确认率、成功率、误判率

## 8. 当前开发应遵守的治理约束

1. 在线运行时收益不得直接写正式语义表。
2. 治理发布必须优先回写 PostgreSQL 正式层。
3. 检索层是正式层的投影，不是最终治理真相。
4. Prompt 和模型配置也属于治理资产，不能继续完全散落在代码和环境变量中。
5. 发布链路必须可审计、可追踪、可回滚。

## 9. 第一阶段建议

### 第一阶段建议实现

- 最小 `governance_candidates`
- 候选状态流
- 最小发布适配器
- 发布作业审计预留
- 如首批发布链已经跨多人协作，再补最小 `release_runs`
- 审核动作审计表

### 第一阶段暂不建议实现

- 完整自动审批
- 复杂多级审批流
- 完整 `evaluation_runs` 产品化
- 候选自动直接发布
