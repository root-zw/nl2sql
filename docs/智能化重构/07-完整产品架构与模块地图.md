## 2. 文档目标

本文档回答三个问题：

- 最终智能化产品应该由哪些层和哪些模块组成
- 当前仓库已经落地了哪些底座，哪些仍未实现

本文档不是单一模块细节文档，而是整个产品的全景地图。

## 4. 当前运行拓扑

从当前仓库和配置看，运行时至少包括以下基础设施：

- 前端
  - Vue 3 + Vite
  - 构建产物 `frontend/dist/`
  - 最终由 `server/main.py` 托管
- 应用服务
  - FastAPI
  - 入口 `server/main.py`
- 元数据库
  - PostgreSQL
  - 存放用户、权限、元数据、规则、历史、同步状态
- 缓存与停止信号
  - Redis
- 检索服务
  - Milvus
- 业务数据库连接
  - 由 `database_connections` 动态管理
  - 实际执行 SQL 的目标库不止一个

当前运维方式以以下文件为准：

- `docker/docker-compose.yml`
- `deploy/systemd/nl2sql-infra.service`
- `deploy/systemd/nl2sql-backend.service`

因此最终产品架构必须同时覆盖：

- 代码模块边界
- 数据对象边界
- 工作台边界
- 运行基础设施边界

## 5. 最终产品定义

最终产品不应只是“自然语言转 SQL 服务”，而应同时是以下六条产品线的组合体：

1. 智能分析执行产品线
2. 智能治理与发布产品线
3. 智能身份、组织与策略产品线
4. 智能观测、诊断与运营产品线
5. 智能平台控制与外部接入产品线
6. 智能安全守卫与持续进化产品线

更具体地说，最终产品应满足：

- 用户可在多轮会话里发起查询、确认理解、批准执行、查看解释、反馈结果
- 系统可把运行时收益沉淀成结构化学习事实
- 数据管理员可把稳定收益发布到正式语义层和检索层
- 不同角色在同一套产品中看到不同工作台和不同权限边界
- 研发和运营可以从日志、trace、查询日志、审计和指标还原完整过程

但从当前系统状态出发，最优实施顺序并不是把这些目标一次性平铺开做。

当前更合理的收敛方式是：

- 先围绕执行域建立权威状态和动作骨架
- 再建立学习事实和治理入口
- 再让权限、观测、平台、安全、交付围绕这些核心对象补齐产品闭环

因此本文件里的“完整地图”应理解为目标态地图，不是第一波实施的并行开工清单。

## 9. 角色与工作面

当前代码里的系统角色仍是：

- `admin`
- `data_admin`
- `user`

更合理的最终产品理解方式是：

- 系统角色
  - 控制后台和高危能力入口
- 数据角色
  - 控制可访问的数据范围
- 工作面角色
  - 控制一个人主要在哪个工作台工作

因此后续不建议把所有职责都堆到 `users.role` 里，而应区分：

- 系统管理员
  - 平台和高危配置
- 数据管理员
  - 元数据、规则、发布、同步
- 审核者
  - 候选审核与发布审批
- 分析用户
  - 日常问数、反馈、偏好确认
- 运维/排障角色
  - 查询日志、trace、健康检查、同步排障

## 10. 目标态完整主流程

以下流程是目标态完整闭环，不等于第一波必须一次性全部显式化到独立对象。

完整主流程建议固定为：

1. 用户登录并进入问数工作台
2. 系统建立会话并创建查询状态对象
3. 系统生成或更新 Query Draft
4. 用户确认、修订或补充上下文
5. 系统执行检索、解析、编译、权限注入和执行
6. 系统返回结果、解释和可选后续动作
7. 系统记录学习事件与运行时记忆
8. 系统对稳定收益生成治理候选
9. 审核者在治理工作台审阅候选
10. 发布适配器创建 `release_runs` 并把通过的候选回写正式语义层
11. 同步链路把正式层变化扩散到 Milvus
12. 发布前后验证沉淀到 `evaluation_runs`
13. 观测层通过 `system_events` 跟踪收益、错误和回滚信号
14. 必要时写入 `rollback_records`
15. 后续查询从正式层、检索层、Prompt 层和模型配置层共同受益

## 11. 后续开发的硬约束

1. 不推翻现有智能分析执行内核，优先做外层产品化重构；当前主体仍是 NL2SQL 相关链路。
2. 在线运行时收益不得直接越权写正式语义层。
3. 每个新增模块都必须回答“当前事实是什么、目标态新增什么、如何与现有代码衔接”。
4. 不把系统角色、数据角色、治理审批职责混成同一种角色模型。
5. 可观测性必须与状态层一起建设，不能等上线后再补。
6. Prompt、规则、检索对象、模型配置都属于可治理资产，不能再散写成隐式配置。
7. 平台配置、缓存、词典和外部适配不能继续散落为隐式运行约定。
8. SQL 安全、成本守护、Dry Run、取消和导出限制必须形成统一运行守卫层。
9. 迁移、初始化 SQL、测试和部署必须按同一交付链路管理，不能脱节演进。
