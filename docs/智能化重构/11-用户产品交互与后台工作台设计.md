## 2. 文档目标

- 当前用户问数和后台操作的实际交互是什么样

## 4. 当前用户问数操作流

### 4.1 当前基础流

当前用户操作基本是：

1. 登录
2. 进入 `/chat`
3. 选择可访问连接
4. 输入问题
5. 通过 `WS /api/query/stream` 发起查询
6. 后端创建消息占位和活跃查询
7. 前端接收进度事件、确认事件、结果事件
8. 用户在必要时确认执行或选择表
9. 会话消息落库并进入历史

### 4.2 当前已存在的确认

当前后端真正稳定支持的确认主要有：

- 高成本执行确认
- 表选择确认

当前还没有稳定落地的部分：

- 完整语义级 Query Draft 修订
- 细粒度字段级澄清
- 反馈驱动的记忆保存界面

## 5. 最终用户工作台设计

### 5.1 分析用户工作台

最终分析用户工作台应至少包含：

- 会话列表
- 当前查询状态
- Query Draft 区
- 澄清与确认区
- 执行结果区
- 结果上下文与追问入口
- 结果解释区
- 反馈与记忆入口

建议把一次查询固定拆成以下面板：

- 用户问题
- 系统当前理解
- 待确认点
- 当前执行状态
- 结果与解释
- 后续建议动作

#### chat-first 对话式承载原则

最终目标态里，前台仍应只有一个以对话为主的问数界面，而不是把“提问页”“确认页”“结果页”“追问页”拆成多个独立流程页。

需要明确：

- 初始提问在对话里发生
- 表选择确认 / Draft 确认 / 执行确认在对话里发生
- 结果出来后的继续分析在对话里发生
- 基于上一份结果再查另一部分并做对比，也仍在同一对话线程里发生

因此最终前台更合适的形态是：

- 一个主聊天流
- 助手消息里按需挂载紧凑的确认卡、结果卡、结果上下文卡
- 按钮只是快捷入口，不应替代自然语言对话本身

### 5.2 用户动作协议

建议最终统一为结构化动作，而不是继续只靠特殊字段：

- `choose_table`
- `change_table`
- `manual_select_table`
- `choose_option`
- `confirm_draft`
- `revise_field`
- `add_context`
- `approve_execution`
- `reject_execution`
- `abandon_and_new_query`
- `give_feedback`
- `remember_preference`
- `cancel_query`

建议同时固定两个交互约束：

- 动作推进后的产品级查询继续复用原 `query_id`
- 如需继续流式输出，可以重建 transport 连接，但不应重建业务级查询标识

建议再固定一个输入框交互规则：

- 当当前 `query_id` 仍有 `pending_actions` 时，用户继续输入自然语言，不默认视为“新问题”
- 前端应先把这条输入交给后端做 `pending_reply_resolution`
- 只有在后端明确返回 `abandon_and_new_query` 时，前端才把它当成新问题开启新的 `query_id`

前端需要支持的三种返回结果应固定为：

- `resolved_to_action`
  - 把用户文本映射成当前 pending 下的正式动作
- `resolved_to_new_query`
  - 显式结束当前等待态，并在同一会话中开启新问题
- `need_clarification`
  - 只展示一轮极短澄清，不直接执行

这意味着最终交互里：

- 按钮只是快捷触发
- 自然语言回复先转动作
- “开始新问题”也属于正式动作语义，而不是输入框的隐式副作用

建议再固定一个“结果完成后继续追问”的输入框规则：

- 当当前 `query_id` 已完成，且同一会话里存在最近可复用的结果上下文时，用户继续输入自然语言，不应立刻假设这是一个完全独立的新问题
- 前端应先把这条输入交给后端做 `followup_context_resolution`
- 该步骤只负责判断这句话是在继续分析上一份结果、基于上一份结果发起对比，还是开启一个独立新问题

前端需要支持的四类结果建议固定为：

- `continue_on_result`
  - 基于上一份结果继续分析
- `compare_with_result`
  - 以已有结果作为基准，再补查另一部分数据并进入对比
- `resolved_to_new_query`
  - 这是一个不依赖上一份结果的独立新问题
- `need_clarification`
  - 系统无法稳定判断是否依赖上一份结果，只追加一句极短澄清

### 5.3 确认策略开关建议

最终用户工作台建议支持确认策略，而不是只有“是否确认”的隐式行为。

建议第一阶段至少支持：

- `adaptive`
  - 默认模式
  - 系统根据置信度、歧义度和风险等级决定是否需要确认
- `always_confirm`
  - 每次提问都先展示统一确认容器

建议交互入口：

- 会话级确认策略切换
- 用户偏好设置中的默认确认策略

### 5.4 确认策略的产品边界

必须明确：

- `confirmation_mode` 只控制语义确认 / 草稿确认
- 不控制高风险执行确认
- 不控制安全保护层的强制确认

因此即使用户选择 `adaptive`：

- 成本超限仍必须确认
- 安全风险仍必须确认
- 强制人工决策点仍必须确认

### 5.5 统一确认组件在不同策略下的行为

#### `adaptive`

- 高置信度、低歧义问题可自动推进
- 低置信度表选择进入 `choose_table`
- 关键歧义 Draft 进入 `confirm_draft`
- 高风险执行进入 `approve_execution`

#### `always_confirm`

- 每次提问都先展示统一确认组件
- 如表未确认，先展示候选表与暂定草稿
- 用户确认表后，系统重算并再次展示 Draft 确认
- 如存在高风险执行，再进入执行批准

#### 待确认阶段继续输入时的产品行为

当统一确认容器仍处于活跃状态时，用户继续在输入框中输入，应遵循以下产品规则：

1. 优先判断这句话是否在回应当前 pending
2. 如判断为当前 pending 的回复，则继续作用于当前 `query_id`
3. 如判断为新问题，则显式结束当前等待态并创建新 `query_id`
4. 如判断不稳定，则追加一句极短澄清

建议默认澄清文案风格固定为：

- “你是想继续刚才这条查询，还是开始一个新问题？”

不建议在这个场景下：

- 静默把所有自由文本都当成新问题
- 静默把所有自由文本都当成当前 pending 的修订
- 在低置信度下直接推进表选择、Draft 确认或执行确认

#### 高置信度自动推进后的纠错入口

即使在 `adaptive` 模式下表选择被判定为高置信度并自动推进，产品层仍必须保留显式纠错入口。

结果区或统一确认容器应始终展示：

- 本次使用的数据表
- `change_table`
- `manual_select_table`

其中：

- `change_table`
  - 语义是“否定当前自动选表并重新规划”
  - 不是“把同样问题原样再问一次模型”
- `manual_select_table`
  - 语义是“跳过本轮选表模型，直接由用户指定表”

#### 最终人工兜底建议

最终表选择纠错建议固定采用以下优先级：

1. 先复用当前候选表快照，让用户直接改选
2. 如需模型重规划，必须带上“已否定当前表”的状态
3. 如仍不稳定，进入定向澄清
4. 最后允许用户从“全部可访问表”中手动选表

用户手动选表后：

- 该选择应直接成为本轮查询真相
- 不再让选表模型覆盖这一步的结果
- 后续 Draft / IR / SQL / result 必须基于用户所选表重新生成

### 5.6 结果完成后的对话记忆与追问

最终产品里需要把“会话历史”与“结果级分析上下文”区分开。

其中：

- 会话历史
  - 负责还原用户和助手说过什么
- 结果级分析上下文
  - 负责告诉系统“当前追问是围绕哪一份结果继续分析”

建议固定以下产品语义：

1. 当用户说“基于刚才这份结果，再看华东区域”时：
   - 系统应识别为 `continue_on_result`
   - 在同一会话中创建新的 `query_id`
   - 新查询继承上一份结果的最小分析上下文，而不是只把整段聊天原文硬塞给模型
2. 当用户说“再查去年同期，并和刚才这份结果对比”时：
   - 系统应识别为 `compare_with_result`
   - 新查询应显式带上上一份结果引用，作为对比基准
   - 后续表选择、IR 生成和解释都要知道“当前是基于已有结果发起的对比型追问”
3. 当用户说的是一个无关新问题时：
   - 系统应创建新的 `query_id`
   - 默认不继承上一份结果上下文
4. 当系统无法稳定判断时：
   - 只追问一句极短澄清
   - 不直接把旧结果强行带入新问题

这部分也必须保持 chat-first：

- 用户继续在同一个输入框里追问
- 助手通过对话消息说明当前理解的结果范围
- 如有需要，再附带“继续分析这份结果”“按这份结果做对比”“清空结果上下文”等快捷按钮

### 5.7 结果对比型追问的展示建议

当系统识别出当前问题依赖上一份结果时，前端建议在对话消息中显式展示当前结果上下文，而不是只在后台隐式保存。

建议展示的最小信息：

- 当前继承的是哪一份结果
- 该结果来自哪条查询
- 当前保留的核心筛选范围
- 当前是否处于“继续分析”还是“结果对比”模式

如果当前进入对比型追问，建议助手先在消息中明确：

- 对比基准是什么
- 即将补查的另一部分数据是什么
- 如上下文不足，直接在对话里补一轮极短澄清

- 关系确认

- 证据对比
- 审批
- 发布
- 回滚

## 8. 目标态完整用户旅程

以下旅程描述的是目标态完整闭环，不等于第一阶段必须把所有对象都显式化成独立表。

建议固定为：

1. 用户进入问数工作台
2. 创建或进入会话
3. 系统创建 `query_sessions` 记录
4. 系统展示 Query Draft 或等价的当前理解视图
5. 用户确认或修订
6. 系统继续在同一 `query_id` 下执行
7. 系统展示结果与解释
8. 用户可继续围绕该结果追问或发起结果对比
9. 系统在同一会话中新建 `query_id` 并继承最小结果上下文
10. 用户反馈对错、补充偏好
11. 系统写学习事件
12. 稳定收益进入候选池
13. 审核者触发 `release_runs`
14. 发布前后验证进入 `evaluation_runs`
15. 必要时进入 `rollback_records`
16. 后续查询受益

其中针对表选择纠错，建议明确：

- 用户执行 `change_table` / `manual_select_table` 后，系统先完成本轮重规划
- 然后再把这次交互写成学习事件
- 只有重复出现、且证据稳定的模式，才应进入记忆或治理候选

## 9. 当前开发硬约束

1. 前台问数工作台不能再只围绕“文本框 + 结果表格”设计。
2. 后台菜单不应再只是技术管理页集合，而应逐步收敛为角色工作台。
3. 用户动作必须逐步结构化，不能长期靠 `force_execute`、`selected_table_id` 这类特殊字段承载所有交互。
4. 每个工作台都要明确：
   - 谁使用
   - 解决什么问题
   - 依赖哪些对象
   - 操作后写哪些状态或审计

## 10. 第一阶段建议

### 第一阶段建议实现

- 在现有 Chat 页中引入最小 Query Draft / 当前理解区
- 在后台新增候选审核页原型
- 在查询日志页新增 trace 跳转
- 在权限页补充权限审计入口
- 让确认后的继续执行沿用原 `query_id` 和用户可见消息

### 第一阶段暂不建议实现

- 大规模重写全部后台页面
- 一开始就做复杂多角色门户
