# 实施矩阵与阶段蓝图

## 1. 文档信息

- 状态：草案
- 类型：系统级实施蓝图文档
- 最后更新时间：2026-04-17

## 2. 文档目标

本文档回答以下问题：

- 当前半成品系统如何逐步改造成最终目标态
- 每个阶段到底改哪些层面，而不是只写方向
- 哪些改造优先，哪些必须作为前置约束
- 如何确保重构不是一组分散 RFC，而是一条可执行的系统演进路线

## 3. 实施原则

### 3.1 最终目标优先

本蓝图以最终目标态为准，不以当前工作量大小为约束。

但“目标优先”不等于“无序推进”，仍然必须有分阶段路径，因为当前系统是半成品：

- 智能能力已分散落地
- 工作台已部分存在
- 数据对象和迁移体系已部分存在
- 直接整体推翻会让现有价值无法复用

### 3.2 先建立骨架，再推进全系统智能化

实施顺序必须坚持：

1. 先建立顶层定义
2. 先建立全局控制平面
3. 先收敛共享契约
4. 再改执行域、治理域、策略域、平台域和工作台
5. 最后把评测、发布、回滚、持续进化完全产品化

### 3.3 每个阶段都必须同时覆盖五个面

任何阶段都不应只改一层：

- 数据对象与迁移
- 后端接口与服务
- 前端与后台工作台
- 测试与观测
- 文档与交付规则

## 4. 当前起点

当前仓库已经有以下可复用基础：

- 智能分析执行内核
  - `server/api/query/`
  - `server/nl2ir/`
  - `server/compiler/`
  - `server/exec/`
- 会话与消息底座
  - `conversations`
  - `conversation_messages`
  - `active_queries`
- 正式治理与同步底座
  - `fields`
  - `field_enum_values`
  - `table_relationships`
  - `global_rules`
  - `server/sync/`
- 身份策略底座
  - `server/auth/`
  - `server/api/admin/permissions.py`
  - `server/api/admin/organizations.py`
- 平台控制底座
  - `server/api/admin/system_config.py`
  - `server/api/admin/cache.py`
  - `server/api/admin/tokenizer.py`
- 可观测底座
  - `server/trace/`
  - `server/api/admin/monitor.py`
- 交付底座
  - `migrations/`
  - `tests/`
  - `docker/docker-compose.yml`
  - `deploy/systemd/`

因此实施路线应以“外层系统化重构 + 内核尽量复用”为基线。

## 4.1 当前最优实施收敛

从当前代码现状出发，阶段蓝图不应被理解成“所有目标态对象都会按阶段逐个立刻显式化”。

更合理的近期收敛是：

- 第一波核心对象优先围绕 `query_sessions`、`draft_actions`、`learning_events`、`governance_candidates`
- 在进入这批对象的真实开发前，先冻结统一确认读模型、Prompt 编排协议和 Skill 分层边界，避免后续边做对象边返工交互与 Prompt 结构
- `query_drafts`、`runtime_memories`、`policy_snapshots`、`evaluation_runs`、`rollback_records`、`system_events` 可以先作为派生读模型、预留门禁或第二阶段对象
- 六条产品线的完整产品化是北极星，不是近期并行开工清单

因此后续各阶段的理解方式应是：

- 先解决权威状态、动作协议、学习事实和治理入口
- 再扩大到策略快照、发布评测、回滚、系统事件等产品化对象
- 最后再把平台控制面和多工作台完全做厚

## 5. 最终目标倒推的阶段路线

### 阶段 0：顶层定义与对象收口

目标：

- 先把整个系统智能化产品的顶层定义、全局控制平面和实施骨架钉住

主要改动：

- 文档层
  - 建立系统级顶层定义
  - 建立全局控制平面文档
  - 建立实施矩阵文档
- 契约层
  - 收敛共享对象与全局标识命名

交付物：

- `00-系统级智能化产品定义与边界.md`
- `00-全局控制平面与闭环设计.md`
- `15-实施矩阵与阶段蓝图.md`
- 重写后的目录导航

完成标准：

- 后续每篇文档都能被挂到新的系统级骨架下
- “整个系统智能化”不再停留在口号层

### 阶段 1：执行域状态骨架重构

目标：

- 先把当前查询域从“隐式过程控制”改造成“显式状态推进”

主要改动：

- 数据对象
  - 新增最小 `query_sessions`
- 后端
  - 拆分 `server/api/query/routes.py` 的节点边界
  - 统一节点输入输出
  - 让取消、恢复、继续执行依赖 `query_sessions`
- 契约
  - 明确 `query_sessions`、`conversation_messages`、`active_queries`、`QueryTracer` 的权威边界
- 前端
  - 先不重写整页
  - 先让 Chat 页能读取和展示统一状态
- 测试
  - 新增 `query_sessions` 迁移测试
  - 新增状态推进回归

完成标准：

- 查询状态不再只散落在 `conversation_messages`、`active_queries`、`tracer`

### 阶段 2：Draft 读模型与动作协议落地

目标：

- 把“系统当前理解”和“用户动作”变成结构化对象

主要改动：

- 数据对象
  - 新增 `draft_actions`
  - 视前端交互复杂度决定是否显式化最小 `query_drafts`
- 后端
  - 兼容现有 `POST /api/query`
  - 新增 Draft 动作接口
  - 统一表选择确认与执行确认的动作表达
  - 先允许 `query_sessions.state_json` 加 `draft_actions` 承接最小草稿读模型
  - 固定动作写入、状态推进与异步派生的事务边界
- 前端
  - 在 `frontend/src/views/Chat.vue` 中引入最小 Draft 区
  - 让确认卡与后续动作统一写回动作协议
- 测试
  - 动作幂等测试
  - 版本冲突测试
  - 表选择/执行确认兼容测试

完成标准：

- `selected_table_id(s)`、`force_execute` 不再是唯一主交互载体
- 即使暂不显式建 `query_drafts`，前端也能通过统一动作协议看到当前理解和待确认项

### 阶段 3：学习、记忆与收益提炼落地

目标：

- 把查询域收益从“样本沉淀”升级成统一学习事实层

主要改动：

- 数据对象
  - 新增 `learning_events`
  - 视收益提炼复杂度新增最小 `runtime_memories`
- 后端
  - 查询完成、反馈、记忆写入统一进入事件层
  - 异步任务提炼运行时记忆
  - 先固定 `learning_events -> governance_candidates` 的主顺序
  - 如需运行时增强，再在中间插入最小 `runtime_memories`
- 前端
  - 增加反馈与“记住这次偏好”入口
- 测试
  - 事件记录回归
  - 记忆写入与过期回归

完成标准：

- few-shot 样本不再承担全部学习闭环职责
- 即使 `runtime_memories` 仍保持最小形态，也已经有统一事件层承接收益事实

### 阶段 4：治理候选与发布平面落地

目标：

- 把运行收益稳定接到正式治理层

主要改动：

- 数据对象
  - 新增 `governance_candidates`
  - 视范围新增 `release_runs`
  - 视范围新增 `rollback_records`
- 后端
  - 增加候选生成器
  - 增加发布适配器
  - 增加 `release_runs` / `rollback_records` 作业入口
  - 为 `evaluation_runs` 固定阻断门禁和回滚触发规则，但允许先不把它做成独立产品化对象
- 后台
  - 增加候选池、审核页、发布页
- 测试
  - 候选到正式层映射测试
  - 同步触发测试
  - 回滚回归测试

完成标准：

- 运行收益可以稳定进入正式层，而不是停留在临时补丁
- 发布门禁已经具备最小可追踪语义，不要求一开始就完成完整评测平台

### 阶段 5：策略平面与系统工作台重构

目标：

- 把身份、组织、权限、能力边界升级成系统级策略平面

主要改动：

- 数据对象
  - 显式化 `policy_snapshots`
- 后端
  - 查询、发布、高危操作统一生成策略快照
  - 权限、导出、安全守卫与审计统一关联
- 后台
  - 权限与组织工作台改造成策略工作台
  - 补权限审计入口
- 测试
  - 策略快照测试
  - 行级过滤与导出控制测试

完成标准：

- 权限不再只是“接口前的判断”，而成为可回放、可审计的系统对象

### 阶段 6：观测、评测、发布、回滚产品化

目标：

- 让系统具备正式的持续进化能力

主要改动：

- 数据对象
  - 显式化 `evaluation_runs`
  - 视范围显式化 `system_events`
- 后端
  - 统一结构验证、行为验证、冒烟验证记录
  - 发布前后验证结果统一落库或集中记录
- 后台
  - 查询日志工作台补全关联对象跳转
  - Trace 工作台产品化
  - 发布收益与回滚工作台产品化
- 测试
  - 发布验证链测试
  - 回滚链测试

完成标准：

- 发布、评测、回滚从工程流程变成产品能力
- 这一步才是 `evaluation_runs`、`rollback_records`、相关工作台全面显式化的主阶段

### 阶段 7：平台控制平面与外部适配统一化

目标：

- 把配置、缓存、词典、事件总线和外部接入统一成平台控制面

主要改动：

- 数据对象
  - 如有必要，补齐 `system_events` 的持久化落点
- 后端
  - 统一配置优先级模型
  - 统一缓存依赖与失效审计
  - 扩展实时事件为系统级事件总线
  - 外部适配统一接入状态、策略、审计和能力开关
  - 固定“配置变更 -> 缓存失效 -> 生效时机”的矩阵
- 后台
  - 平台控制台成型
- 测试
  - 配置热加载测试
  - 缓存失效测试
  - 外部接入约束测试

完成标准：

- 平台域不再只是多个运维接口集合，而是系统控制平面

## 6. 各阶段统一改造矩阵

每个阶段都必须回答以下问题：

- 数据库需要新增哪些表、字段、索引、迁移
- 后端需要新增哪些服务、接口、异步任务
- 前端或后台需要新增哪些工作台、卡片、动作入口
- 测试需要覆盖哪些 unit、schema、integration、smoke
- 发布前验证什么
- 回滚点在哪里

如果有一项答不上来，该阶段设计就还不完整。

### 6.1 施工附录承接方式

本文件负责回答“阶段顺序和阶段目标”。

当进入直接施工阶段时，还必须同时查看：

- `16-施工附录与代码落点清单.md`
  - 给出每个阶段建议修改的真实代码文件、迁移落点、前端页面、测试文件、验收动作和回滚点

因此后续实现时的职责分工应固定为：

- `15`
  - 决定先做什么、后做什么
- `16`
  - 决定具体改哪里、怎么验收、哪里可回滚

## 8. 当前开发硬约束

1. 任何阶段都不能只改一端而不改共享契约、测试和实施说明。
2. 任何新增核心对象都必须同时评估 Alembic、初始化 SQL、后台工作台和回归测试。
3. 任何智能化收益都必须回答“进入运行时层还是正式治理层”。
4. 任何系统级工作台都必须回答“写哪些状态、事件、审计和评测对象”。
5. 如果某项改造无法被发布、评测和回滚链路承接，它就还不能算真正落地。
