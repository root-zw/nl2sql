# 平台支撑层与外部集成设计

## 1. 文档信息

- 状态：草案
- 对应总文档章节：`00-智能化重构总方案.md`
- 最后更新时间：2026-04-16

## 2. 文档目标

本文档回答以下问题：

- 当前系统有哪些平台支撑能力已经落地
- 哪些能力不是“问数主链路”，但决定了整个产品能否长期运转
- 最终智能化产品还需要补哪些平台级模块
- 外部系统如何接入，而不绕过权限、审计和状态层

## 3. 当前现实

### 3.1 当前已经存在的平台支撑面

当前系统并不只有查询和治理接口，还已经有一批平台级支撑能力：

- 系统配置与热加载
  - `server/config.py`
  - `server/api/admin/system_config.py`
- 缓存与模型装载
  - `server/exec/cache.py`
  - `server/metadata/db_manager.py`
  - `server/api/admin/cache.py`
- Prompt / 规则 / 模型配置缓存
  - `server/utils/prompt_loader.py`
  - `server/utils/global_rules_loader.py`
  - `server/nl2ir/llm_client.py`
- 分词器与词典
  - `server/nl2ir/tokenizer.py`
  - `server/api/admin/tokenizer.py`
  - `config/domain_dict.txt`
- 实时事件与 WebSocket
  - `server/api/admin/websocket.py`
  - `server/websocket_manager.py`
- 外部系统适配
  - `server/api/dify_adapter.py`
  - `server/api/admin/user_sync.py`
- 健康探针与服务壳
  - `server/api/health.py`
  - `server/frontend_static.py`

因此最终架构不能只把系统理解为“一个查询 API 加一个后台”，还必须把这些平台能力单独建模。

### 3.2 当前配置控制面的真实情况

当前配置来源并不只有一种：

- `.env` 与环境变量
  - 由 `server/config.py` 统一读取
- `config/retrieval_config.yaml`
  - 由 `server/config.py` 读取
  - 支持 `reload_retrieval_config()`
- 数据库里的模型与 Prompt 配置
  - `model_providers`
  - `provider_models`
  - `scenario_model_configs`
  - `prompt_templates`

当前已经有的后台接口：

- `/api/admin/system-config/model`
- `/api/admin/system-config/retrieval/info`
- `/api/admin/system-config/retrieval/reload`

这说明当前已经有“配置控制面”的雏形，但还没有被单独抽象成平台层。

### 3.3 当前缓存与热加载面的真实情况

当前缓存不是一处，而是多处并存：

- 查询结果缓存
  - `server/exec/cache.py`
  - Redis Key 基于 IR 和用户上下文生成
- 语义模型缓存
  - `server/metadata/db_manager.py`
- 规则缓存
  - `server/utils/global_rules_loader.py`
- Prompt 缓存
  - `server/utils/prompt_loader.py`
- LLM 场景配置缓存
  - `server/nl2ir/llm_client.py`
- 分词缓存
  - `server/nl2ir/tokenizer.py`

当前后台已经有的运维入口：

- `/api/admin/cache/invalidate/{connection_id}`
- `/api/admin/cache/invalidate-all`
- `/api/admin/cache/stats`
- `/api/admin/cache/reload/{connection_id}`

因此当前系统已经存在明显的“缓存平面”，只是还缺统一依赖关系和失效审计。

### 3.4 当前词法与中文处理支撑面

当前检索和关键词处理并不完全依赖向量检索，还依赖统一分词器：

- 分词核心
  - `server/nl2ir/tokenizer.py`
- 稀疏检索与关键词提取会复用分词器
  - `server/nl2ir/sparse_utils.py`
  - `server/nl2ir/keyword_pipeline.py`
  - `server/nl2ir/hierarchical_retriever.py`
- 管理接口
  - `/api/admin/tokenizer/stats`
  - `/api/admin/tokenizer/tokenize`
  - `/api/admin/tokenizer/cache/clear`
  - `/api/admin/tokenizer/dictionary/reload`
  - `/api/admin/tokenizer/dictionary/add-word`

这意味着词典、分词模式和缓存策略本身也是平台级运行资产，而不是单纯算法细节。

### 3.5 当前实时事件与后台通信面

当前已经有一套实时通信面，但主要服务于同步系统：

- WebSocket 接口
  - `server/api/admin/websocket.py`
- 连接管理与广播
  - `server/websocket_manager.py`
- 主要事件
  - 同步开始
  - 同步进度
  - 同步完成
  - 健康更新
  - 待同步变更更新

当前这套能力还没有扩展到：

- 查询状态广播
- Draft 动作广播
- 治理候选广播
- 发布结果广播

### 3.6 当前外部适配与产品壳

当前已经有两类明确的外部接入面：

- Dify 工具适配
  - `server/api/dify_adapter.py`
  - 会把 Dify 请求转换为统一 `QueryRequest`
  - 且在 Dify 场景下关闭内部叙述生成
- 外部用户同步
  - `server/api/admin/user_sync.py`
  - 支持按认证提供者同步用户、数据角色和用户属性

当前还有一层产品壳能力：

- 前端静态产物由后端托管
  - `server/frontend_static.py`
- 前端标题和品牌有环境变量入口
  - `frontend/src/router/index.js`
  - `frontend/src/views/Chat.vue`

### 3.7 当前健康检查与服务能力探针

当前系统已经有：

- `/health`
  - 检查数据库与 Redis
- `/ready`
  - 提供最小 readiness probe

但当前还没有形成更完整的能力探针，例如：

- Milvus 可用性
- Embedding / Reranker 可用性
- Auto Sync 服务状态
- Prompt / 模型配置加载状态

## 4. 当前遗漏的系统级架构面

如果从最终产品角度看，前面文档还缺少以下平台支撑域：

- 配置控制面
  - 配置来源优先级、热加载和回滚
- 缓存控制面
  - 哪些缓存存在、何时失效、谁负责刷新
- 词典与词法控制面
  - 分词、词典、关键词抽取与稀疏检索参数
- 实时事件总线
  - 查询、治理、同步、运维事件的统一广播
- 外部适配网关
  - Dify、外部认证、用户同步和未来 Agent / BI 接入
- 健康与能力探针
  - 降级能力、可用性声明和故障边界

## 5. 目标态平台支撑架构

### 5.1 配置控制平面

最终建议把配置拆成三层：

- 环境级配置
  - 基础地址、端口、凭证注入、部署模式
- 产品级正式配置
  - 模型供应商、场景模型、Prompt、检索参数
- 运行时缓存视图
  - 当前进程已经加载的生效版本和缓存版本

这层至少要明确：

- 配置来源优先级
- 配置生效范围
- 热加载入口
- 回滚入口
- 变更审计

建议后续不要把所有配置硬塞进同一优先级链，而是按类型固定：

- 部署级基础配置
  - 环境变量优先
  - 代码默认值兜底
- 产品级正式配置
  - 数据库正式配置优先
  - 文件基线配置兜底
  - 代码默认值最后兜底
- 应急降级或只读开关
  - 可允许环境变量临时覆盖
  - 但必须进入审计与运维记录

### 5.2 缓存控制平面

最终应把缓存显式分层：

- 查询结果缓存
- 语义模型缓存
- 规则缓存
- Prompt 缓存
- 模型配置缓存
- 分词缓存

每层都应回答：

- Key 是什么
- 依赖哪些正式对象
- 谁触发失效
- 如何观测命中率
- 是否与权限上下文绑定

建议第一阶段至少形成以下失效矩阵：

- 正式元数据变更
  - 影响语义模型缓存、检索消费视图与必要的查询结果缓存命名空间
- `global_rules` 变更
  - 影响规则缓存与依赖规则的在线链路
- `prompt_templates` 变更
  - 影响 Prompt 缓存与对应场景加载视图
- `scenario_model_configs` / 模型供应商变更
  - 影响模型配置缓存与场景客户端视图
- 词典与分词参数变更
  - 影响分词缓存、稀疏检索行为与关键词处理链路
- `retrieval_config.yaml` 变更
  - 影响检索参数视图，并应评估是否需要切换查询结果缓存版本

### 5.3 词法与 NLP 支撑平面

最终不应把分词器仅当作算法内部实现，而应作为平台资产管理：

- 词典版本
- 行业或业务词增量
- 稀疏检索分词参数
- 关键词提取策略
- 分词缓存
- 词典热更新审计

### 5.4 实时事件平面

建议把当前同步 WebSocket 扩成统一事件总线，但第一阶段仍可基于现有实现演进：

- 查询事件
  - 由 `query_sessions`、`draft_actions` 驱动创建、确认、取消、完成
- 治理事件
  - 由 `governance_candidates`、`release_runs`、`rollback_records` 驱动候选生成、审核、发布、回滚
- 同步事件
  - 排队、运行、失败、完成
- 运维事件
  - 通过 `system_events` 记录健康告警、降级、恢复

### 5.5 外部适配网关

外部系统接入不应直接拼内部函数，而应统一走适配层：

- Dify 工具适配
- 外部用户同步
- 未来 BI / 工具平台接入
- 未来多 Agent 或流程平台接入

统一约束建议是：

- 必须走统一 QueryRequest / 动作协议
- 不能绕过权限与审计
- 不能绕过状态层与查询日志
- 需要独立限流和能力开关

### 5.6 健康与能力探针平面

最终应能明确回答：

- 系统是否可登录
- 是否能执行查询
- 是否能访问 Redis / PostgreSQL / Milvus
- 模型调用是否可用
- 同步服务是否正常
- 当前处于正常、降级还是只读保护模式

## 6. 平台支撑层如何影响主链路

### 6.1 配置变化如何进入主链路

当前已经存在的真实路径包括：

- 检索参数
  - `server/config.py` 读取 `config/retrieval_config.yaml`
- Prompt
  - `server/utils/prompt_loader.py`
- 模型场景配置
  - `server/nl2ir/llm_client.py`
- 规则
  - `server/utils/global_rules_loader.py`

因此后续不能只写“改下 Prompt / 改下配置”，而必须写清：

- 改的是哪一层正式配置
- 哪些缓存需要失效
- 哪些后续请求才会看到新效果

建议未来把“何时生效”也固定写清：

- 立即热加载生效
- 仅新请求生效
- 需要手动失效缓存后生效
- 需要发布或重启后生效

### 6.2 词典变化如何影响检索

词典和分词配置会直接影响：

- 关键词抽取
- 稀疏召回
- 表字段匹配
- 业务域检测

因此词典更新本质上是“检索行为变更”，不应长期停留在手工调词级别。

### 6.3 外部调用如何进入统一产品

以 Dify 为例，当前已经证明一件事：

- 外部工具调用也可以复用统一查询入口

最终更合理的目标是：

- 所有外部调用都进入统一产品状态对象
- 外部调用至少能关联 `query_sessions` 与 `policy_snapshots`
- 统一记录 trace、查询日志、权限结果和审计

## 7. 当前开发硬约束

1. 外部适配器不能绕过查询状态、权限检查、查询日志和审计。
2. 配置优先级必须明确，不允许长期依赖“代码里猜当前生效配置”。
3. 缓存失效、配置热加载和词典热更新必须可追踪、可观测。
4. 分词器、词典和检索参数属于平台资产，不应继续只按调试脚本思路维护。
5. 健康检查不能只回答“进程还活着”，最终要回答“哪些能力还能用”。

## 8. 第一阶段建议

### 第一阶段建议实现

- 平台支撑层对象地图
- 配置来源优先级说明和热加载影响面说明
- 缓存依赖关系与失效触发矩阵
- 把同步 WebSocket 事件面扩展到查询 / 治理事件预留
- 为 `system_events` 预留统一事件契约

### 第一阶段暂不建议实现

- 自建复杂消息中间件平台
- 一次性重写所有缓存机制
- 一开始就做通用插件平台
