# 主链路编排与状态机设计

## 1. 文档信息

- 状态：草案
- 对应总文档章节：`00-智能化重构总方案.md`
- 最后更新时间：2026-04-17

## 2. 设计目标

本模块要解决的问题不是“先选什么框架”，而是先回答：

- 当前一次查询到底经历了哪些阶段
- 当前状态真实存在哪里
- 用户确认、取消、继续执行时，系统靠什么推进

## 3. 当前现实

### 3.1 当前在线入口

当前主查询链路由以下模块共同承担：

- `server/main.py`
- `server/api/query/routes.py`
- `server/api/query/table_selection.py`
- `server/nl2ir/parser.py`
- `server/query_complexity/router.py`
- `server/compiler/compiler.py`
- `server/exec/showplan.py`
- `server/exec/executor.py`
- `server/services/conversation_service.py`
- `server/trace/tracer.py`

### 3.2 当前协议与状态载体

当前查询接口实际是：

- `POST /api/query`

当前流式查询和中途状态推进主要依赖：

- `QueryRequest`
- `QueryResponse`
- `conversation_messages`
- `active_queries`
- `tracer`

当前已经稳定在线工作的中间状态主要是：

- `table_selection_needed`
- `confirm_needed`

### 3.3 当前已实现能力

当前系统已经具备的编排基础包括：

- 表选择确认
- 高成本执行确认
- 查询取消
- 会话消息占位与结果回写
- 复杂查询拆分与 DAG 执行
- few-shot 成功样本写回

### 3.4 当前关键缺口

当前主链路仍然缺少：

- 产品级唯一状态对象
- 用户动作统一协议
- 语义级草稿生命周期
- 可靠的恢复和继续执行语义

## 4. 当前状态应该如何理解

### 4.1 当前已存在的状态承载体

当前状态并不是完全没有，只是分散：

- `conversation_messages`
  - 承担消息存档和结果展示状态
- `active_queries`
  - 承担运行中、取消中、已完成等执行状态
- `tracer`
  - 承担调试追踪和部分续接能力

### 4.2 当前不存在的对象

以下对象当前代码和 SQL 中都还没有：

- `query_sessions`
- `query_state_snapshots`

因此这两个名字在本文中都只能表示目标态建议，不能表示现状。

## 5. 目标设计建议

### 5.1 建议新增的主状态对象

建议未来引入：

- `query_sessions`

建议它承担：

- 查询生命周期主状态
- 当前节点
- 当前错误
- 与草稿、消息、执行引用之间的绑定关系

建议最小字段如下：

- `query_id`
- `conversation_id`
- `message_id`
- `user_id`
- `status`
- `current_node`
- `state_json`
- `last_error`
- `created_at`
- `updated_at`

### 5.2 建议的第一版状态枚举

建议第一版统一以下状态：

- `received`
- `context_loaded`
- `drafting`
- `awaiting_user_action`
- `approved`
- `compiling`
- `awaiting_execution_approval`
- `executing`
- `completed`
- `failed`
- `cancelled`

这里强调两点：

- 这些状态当前尚未落地
- 当前只应作为设计建议，不能视为线上真实枚举

### 5.3 状态权威关系建议

如果后续新增 `query_sessions`，必须同时明确它与当前对象的权威边界：

- `query_sessions`
  - 未来唯一承担“产品级生命周期状态与继续执行真相”
- `conversation_messages`
  - 继续承担用户可见时间线和结果展示投影
- `active_queries`
  - 只承担运行中 / 取消中的快速索引，不再作为恢复真相
- `QueryTracer`
  - 继续承担技术 trace 与调试回放，不参与产品状态推进判定
- `query_history`
  - 继续承担结果归档，不反向驱动状态机

这意味着第一阶段如果引入 `query_sessions`，就不能再允许多个对象同时定义“当前到底执行到哪里”。

### 5.4 第一阶段建议固化到 `state_json` 的信息

若第一阶段仍以最小对象落地，建议至少固化：

- 当前节点输入摘要
- 已解析连接与表
- 当前待确认动作类型
- 最近一次用户动作摘要
- 成本守护与批准结果
- 取消、失败或恢复原因
- 当前执行尝试号或续接次数

## 6. 建议的节点拆分

### 6.1 同步节点

建议先把当前大总控拆出以下稳定节点：

- `receive_query`
- `resolve_connection_and_tables`
- `load_context`
- `parse_or_build_draft`
- `route_execution`
- `compile_and_guard`
- `execute`

### 6.2 异步节点

建议后置为异步的节点包括：

- `write_learning_events`

- `emit_governance_candidates`

## 8. 恢复、幂等与取消建议

### 8.1 当前现实

当前恢复语义是分裂的：

- 调试续接主要由 `tracer` 负责
- 运行状态主要由 `active_queries` 负责
- 消息态主要由 `conversation_messages` 负责

### 8.2 目标态建议

建议未来拆成两层：

- 调试与可读回放
  - 继续由 `tracer` 承担
- 产品级状态推进
  - 改由 `query_sessions` 承担

### 8.3 幂等建议

建议未来至少做到：

- 初始查询以 `query_id` 为主键
- 用户动作以 `action_id` 或幂等键去重
- 学习事件以 `event_key` 去重

### 8.4 流式续接建议

当前系统已经有 `WS /api/query/stream`、`message_id` 停止信号和 `original_query_id` 续接兼容逻辑，因此目标态必须明确以下规则：

- 同一条产品级查询在“提问 -> 确认 -> 继续执行 -> 完成”全过程中应保持同一个 `query_id`
- `message_id` 继续代表同一条用户可见 assistant 消息，不应因为传输重连随意重建
- WebSocket 连接 ID、流式连接对象和 transport 级重试都不属于共享业务标识
- 用户确认或修订后，如果需要继续流式输出，可以重建传输连接，但不应重建产品级 `query_id`
- `original_query_id` 只应作为兼容过渡字段，待动作协议稳定后不再承担主语义

### 8.5 取消语义建议

第一阶段应把取消分成两层：

- 运行层取消
  - 继续通过停止信号或运行中断机制尽快终止当前执行
- 产品层取消
  - 统一由 `query_sessions.status='cancelled'` 与状态摘要固化结果

否则会继续出现“运行已经停了，但产品状态、消息状态和 trace 结束语义不一致”的问题。

## 9. 分阶段落地建议

### 第一阶段

- 不引入新框架
- 先把现有总控拆成更清晰的内部节点
- 先把取消、恢复、继续执行语义梳理清楚

### 第二阶段

- 新增最小 `query_sessions`
- 让运行态和产品态先有统一状态来源

### 第三阶段

- 先接入 `draft_actions`
- 先把表选择确认和执行批准纳入统一动作协议
- 如现有读模型不足，再补最小 `query_drafts`

## 10. 当前不建议直接定案的项

以下内容当前不建议写成定论：

- 一开始就绑定 `LangGraph`
- 一开始就建设完整多 Agent 编排
- 一开始就建设完整快照系统

更务实的顺序仍然是：

- 先抽边界
- 再定状态对象
- 最后评估是否框架化
