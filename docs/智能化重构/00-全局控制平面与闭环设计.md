# 全局控制平面与闭环设计

## 1. 文档信息

- 状态：草案
- 类型：系统级顶层控制文档
- 最后更新时间：2026-04-17

## 2. 文档目标

本文档回答以下问题：

- 为什么当前系统虽然有很多能力，但仍然没有形成统一产品
- 整个系统最终需要哪些全局控制平面
- 各域之间如何通过共享对象、共享标识和统一闭环连接起来
- 哪些对象需要成为后续共享契约和实施蓝图的重点

## 3. 当前现实

### 3.1 当前控制信息分散在哪里

当前系统已经有大量控制信息，但分散在不同对象和模块中：

- 查询状态与消息
  - `conversation_messages`
  - `active_queries`
  - `server/services/conversation_service.py`
- 查询技术追踪
  - `server/trace/tracer.py`
  - `server/trace/api.py`
- 查询历史与结果归档
  - `query_history`
  - `server/api/query/history.py`
- 权限与组织决策依据
  - `users`
  - `organizations`
  - `data_roles`
  - `role_table_permissions`
  - `role_row_filters`
  - `user_data_roles`
  - `org_data_roles`
  - `user_attributes`
- 正式治理与同步控制
  - `metadata_change_log`
  - `milvus_pending_changes`
  - `milvus_sync_history`
  - `sync_errors`
- 模型与 Prompt 正式配置
  - `model_providers`
  - `provider_credentials`
  - `provider_models`
  - `scenario_model_configs`
  - `prompt_templates`
  - `prompt_template_history`
- 平台配置与缓存控制
  - `server/config.py`
  - `config/retrieval_config.yaml`
  - `server/api/admin/system_config.py`
  - `server/api/admin/cache.py`
- 发布与交付控制
  - `migrations/`
  - `tests/`
  - `docker/docker-compose.yml`
  - `deploy/systemd/`

### 3.2 当前为什么还不能叫统一控制平面

因为这些控制信息目前仍然存在以下问题：

- 同一条业务流程需要跨多个对象才能还原
- 系统缺少统一状态主对象
- 系统缺少统一事件主对象
- 权限、治理、发布、评测之间缺少统一关联键
- 后台工作面更多是模块菜单，不是统一闭环工作台
- 发布、回滚和评测尚未成为产品主对象

### 3.3 当前最关键的控制平面判断

从当前代码现状出发，控制平面文档不应被理解成“所有平面都要同优先级同时显式化”。

当前更合理的近期收敛是：

- 先用 `query_sessions` 收敛生命周期状态平面
- 再用 `draft_actions` 收敛动作平面
- 再用 `learning_events` 收敛学习事实平面
- 最后用 `governance_candidates` 打通正式治理入口

以下对象更适合作为第二阶段增强、派生读模型或产品化对象：

- `query_drafts`
- `runtime_memories`
- `policy_snapshots`
- `system_events`
- `release_runs`
- `evaluation_runs`
- `rollback_records`

因此本文件里的“全局控制平面”应理解为目标态完整地图，不是第一阶段并行开工清单。

## 4. 目标态全局控制平面

### 4.1 身份与上下文平面

职责：

- 统一用户、组织、系统角色、数据角色、用户属性
- 为查询、治理、发布、回滚生成一致的身份上下文

当前现实依据：

- `server/auth/`
- `server/middleware/auth.py`
- `server/api/admin/permissions.py`
- `server/api/admin/organizations.py`

目标态建议新增或显式化：

- `policy_snapshots`
  - 固化一次查询、发布或高危操作当时命中的策略与权限上下文

`policy_snapshots` 更适合作为第二阶段显式化对象；第一阶段可先由权限决策摘要与审计关联承接。

### 4.2 生命周期状态平面

职责：

- 统一承载一次查询或后续工作项的产品级状态
- 避免继续把状态散落在消息、运行表、trace 和内存里

当前现实依据：

- `conversation_messages`
- `active_queries`
- `server/trace/tracer.py`

目标态建议新增或显式化：

- `query_sessions`
- 后续可扩展到其他域的运行对象
  - `release_runs`
  - `evaluation_runs`
  - `rollback_records`

当前更适合第一波优先落地的是：

- `query_sessions`

`release_runs`、`evaluation_runs`、`rollback_records` 更适合作为第二阶段到产品化阶段运行对象。

### 4.3 动作与事件平面

职责：

- 把用户动作、系统动作、审核动作、发布动作统一结构化
- 让跨域收益可以被记录、回放、评估和追责

当前现实依据：

- `original_query_id`
- `selected_table_id`
- `selected_table_ids`
- `force_execute`
- `query_history`
- `metadata_change_log`

目标态建议新增或显式化：

- `draft_actions`
- `learning_events`
- `system_events`

其中：

- `draft_actions`
  - 聚焦查询域动作
- `learning_events`
  - 聚焦学习与收益事实
- `system_events`
  - 聚焦跨域系统事件，如发布、回滚、降级、恢复

当前更适合第一波优先落地的是：

- `draft_actions`
- `learning_events`

`system_events` 更适合作为第二阶段跨域事件对象；第一阶段可以先由现有日志、trace 和关键对象关联承接。

### 4.4 运行时收益平面

职责：

- 承载对下一次交互真正有直接增益的运行时收益
- 与正式治理层明确分层

当前现实依据：

- `qa_few_shot_samples`
- `qa_few_shot_feedback`
- `query_history`
- `server/nl2ir/few_shot_writer.py`

目标态建议新增或显式化：

- `runtime_memories`

`runtime_memories` 更适合作为第二阶段异步提炼层，而不是第一阶段和状态层、治理层并列扩张。

### 4.5 正式治理与发布平面

职责：

- 把运行收益转成候选
- 把候选转成审核、发布、回滚和正式层写入

当前现实依据：

- `fields`
- `field_enum_values`
- `table_relationships`
- `global_rules`
- `metadata_change_log`
- `milvus_pending_changes`
- `milvus_sync_history`

目标态建议新增或显式化：

- `governance_candidates`
- `release_runs`
- `rollback_records`

当前更适合第一波优先落地的是：

- `governance_candidates`

`release_runs`、`rollback_records` 更适合作为治理产品化阶段对象。

### 4.6 评测与验收平面

职责：

- 统一记录一次变更上线前后的验证结果
- 让“发布是否带来收益”可被量化，而不是凭感觉

当前现实依据：

- `tests/unit/`
- `tests/schema/`
- `tests/integration/`
- `deploy/systemd/nl2sql-backend.service`

目标态建议新增或显式化：

- `evaluation_runs`
  - 记录结构验证、行为验证、冒烟验证、上线后观察结果

当前更适合第一波优先落地的是：

- 最小验证门禁记录语义

`evaluation_runs` 作为独立对象更适合在发布评测产品化阶段显式化。

### 4.7 可观测与审计平面

职责：

- 统一承接产品状态、技术 trace、运营指标和审计对象

当前现实依据：

- `server/trace/`
- `server/api/admin/monitor.py`
- `permission_audit_log`
- `metadata_change_log`
- `milvus_sync_history`
- `sync_errors`

目标态建议新增或显式化：

- `policy_snapshots`
- `system_events`
- `release_runs`
- `evaluation_runs`
- `rollback_records`

这些对象在第一阶段更适合先通过：

- `query_id` 关联
- 现有日志与 trace
- 查询日志与治理审计视图

来承接最小观测与审计能力，而不是立即全部扩成独立产品对象。

## 5. 目标态共享标识建议

为了让全系统智能闭环可回放、可审计、可关联，后续共享契约至少应统一以下标识：

- `query_id`
- `conversation_id`
- `message_id`
- `draft_id`
- `action_id`
- `event_id`
- `candidate_id`
- `policy_snapshot_id`
- `evaluation_run_id`
- `release_run_id`
- `rollback_id`

其中当前已经较稳定存在的标识主要是：

- `query_id`
- `conversation_id`
- `message_id`

其余标识属于目标态建议，需在共享契约文档中继续收敛。

当前第一波更值得优先稳定的共享标识是：

- `query_id`
- `conversation_id`
- `message_id`
- `action_id`
- `event_id`
- `candidate_id`

`draft_id`、`policy_snapshot_id`、`evaluation_run_id`、`release_run_id`、`rollback_id` 更适合作为第二阶段到产品化阶段标识。

## 6. 目标态完整全局闭环

以下描述的是目标态完整闭环，不等于第一阶段必须把所有对象都显式化到独立表。

### 6.1 查询与学习闭环

建议固定为：

1. 用户发起查询
2. 系统建立状态对象
3. 系统加载身份与策略上下文
4. 系统生成 Draft 或等价的当前理解视图并等待动作
5. 系统执行并记录技术 trace
6. 系统写事件并按需提炼运行时收益
7. 稳定收益转成治理候选
8. 审核后通过发布适配器写正式层
9. 后续查询继续受益

### 6.2 系统配置与发布闭环

建议固定为：

1. 配置、策略或语义资产发生变更
2. 变更进入候选或正式发布流
3. 发布前执行评测与验证，并记录 `evaluation_runs`
4. 发布写正式层并记录 `release_runs`
5. 同步链路扩散到检索层和在线消费层
6. 观测层通过 `system_events` 跟踪收益、错误和回滚信号
7. 必要时回滚并写 `rollback_records`

### 6.3 当前更适合先固定的最小闭环

如果只保留当前最值得优先落地的控制平面闭环，建议先固定：

1. 用户发起查询
2. 写入或更新 `query_sessions`
3. 通过 `draft_actions` 承接确认、修订、继续执行
4. 执行完成后写入 `learning_events`
5. 由事件窗口异步观察 `governance_candidates`
6. 发布仍优先回写现有正式层

交付侧建议先固定：

1. 文档、迁移、代码、测试按同一顺序推进
2. 用最小验证门禁记录阻断发布
3. 用最小发布审计记录承接上线动作
4. 观察窗口和回滚判定先形成规则，再逐步显式化对象

## 7. 当前开发硬约束

1. 任何新能力都不能只在本域内部自洽，还必须回答它属于哪个全局控制平面。
2. 任何新工作台都不能只展示数据，还必须回答它写哪些状态、事件或审计对象。
3. 任何运行时收益都不得直接跳过治理与发布平面写正式层。
4. 任何外部适配器都不得绕过身份、策略、审计和状态平面。
5. 任何发布与回滚都应最终有结构化对象，而不是只靠日志和人工记忆。
6. 控制平面文档必须同时给出目标态完整闭环和当前最优收敛线，不能只写其中一半。
