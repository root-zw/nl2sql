# 共享契约与关键对象定义

## 1. 文档信息

- 状态：草案
- 类型：系统级共享契约文档
- 最后更新时间：2026-04-17

## 2. 文档目标

本文档用于统一 `docs/智能化重构/` 中跨域共享对象、共享标识、最小字段和兼容规则。

但需要先区分两类对象：

- 当前仓库已经存在、可在代码和 SQL 中找到依据的共享对象
- 目标态建议新增、但当前尚未实现的系统级对象

本文档统一的是“命名与最小契约”，不代表这些对象已经落库。

### 6.1 目标态对象权威关系建议

为避免后续新增对象后再次形成双真相，建议固定：

- `query_sessions`
  - 查询生命周期与可恢复状态真相
- `query_drafts`
  - 如显式化后，承担用户可见当前理解真相
  - 第一阶段也可由 `query_sessions.state_json` 与 `draft_actions` 派生读模型承接
- `draft_actions`
  - 产品动作时间线真相
- `analysis_context`
  - 会话内围绕上一份结果继续分析或做结果对比的短期上下文真相
  - 第一阶段可先内嵌在 `query_sessions.state_json`，不必单独建表
- `learning_events`
  - 学习事实真相
- `runtime_memories`
  - 如显式化后，承担下一次问答可直接消费的运行时增强真相
- `governance_candidates`
  - 待审治理建议真相

以下对象继续保留，但不应再承担目标态主真相职责：

- `conversation_messages`
  - 用户消息与结果展示投影
- `active_queries`
  - 运行中快速索引
- `query_history`
  - 查询归档
- `QueryTracer`
  - 技术 trace

### 7.1.1 `analysis_context`

建议它只承载“本次新查询为何依赖上一份结果”的最小真相，不重复保存完整结果数据。

建议最小字段：

- `context_mode`
  - `none` / `followup` / `compare`
- `inherit_from_query_id`
  - 当前查询继承自哪一条历史查询
- `base_result_refs`
  - 当前继续分析所依赖的结果引用
- `comparison_result_refs`
  - 当前做结果对比时的基准结果引用
- `scope_summary`
  - 当前继承了哪些核心范围，例如时间、对象、指标、维度
- `carry_over_flags`
  - 是否继承表、过滤条件、指标、排序等

建议边界固定为：

- `analysis_context` 只服务同一会话内的短期追问
- 它不等于完整聊天原文
- 它也不等于长期 `runtime_memories`
- 它保存的是结果引用和范围摘要，不是再次复制整份结果表

### 7.1.2 `result_ref`

为避免后续追问完全依赖自然语言回忆，建议统一一份最小结果引用结构。

建议最小字段：

- `query_id`
- `message_id`
- `result_source`
  - `conversation_message` / `query_history` / 其他结果快照来源
- `table_ids`
- `metric_ids`
- `dimension_ids`
- `filter_summary`
- `result_summary`
- `row_count`

建议约束：

- `analysis_context` 中只保存 `result_ref` 与摘要，不直接内嵌大结果集
- 真正的结果数据仍以消息结果或查询归档为准
- 如果引用结果已不可用，应降级为摘要理解或补一轮澄清

对于表选择纠错交互，适合沉淀为运行时记忆的内容包括：

- `table_rejection_pattern`
  - 某类问法下某张表经常被用户否定
- `preferred_table_for_intent`
  - 某类问法下用户或组织更常改选为另一张表
- `clarification_preference`
  - 某类歧义下更适合优先澄清而非继续猜表

需要明确边界：

- 单次 `change_table` 不应直接写成长期记忆
- 只有重复出现、且证据足够稳定时，才应提升为 `runtime_memories`
- `runtime_memories` 只作为后续 Prompt / 排序 / 交互策略的增强输入

对于表选择纠错交互，以下重复模式更适合聚合为治理候选：

- 表描述误导，导致用户反复否定自动选表
- 表标签 / 同义词缺失，导致用户总是改选到另一张表
- 业务阶段语义混淆，例如“报批 / 批复 / 现状 / 成交”长期互相误判
- 某张表缺少足够可解释的字段预览，导致用户只能依赖手动选表

也就是说：

- 一次纠错先进入 `learning_events`
- 重复模式再进入 `runtime_memories`

### 7.12 第一波核心对象 DDL 草案

以下 DDL 不是当前事实，而是基于当前仓库建表风格给出的第一波核心对象施工起点。

建议直接沿用当前仓库风格：

- 主键优先 `UUID`
- `jsonb` 字段统一使用 `_json`
- 时间字段统一使用 `TIMESTAMP WITH TIME ZONE`
- `updated_at` 继续复用现有 `update_updated_at_column()` 触发器

```sql
CREATE TABLE IF NOT EXISTS query_sessions (
    query_id UUID PRIMARY KEY,
    conversation_id UUID REFERENCES conversations(conversation_id) ON DELETE SET NULL,
    message_id UUID REFERENCES conversation_messages(message_id) ON DELETE SET NULL,
    user_id UUID NOT NULL REFERENCES users(user_id) ON DELETE CASCADE,
    status VARCHAR(32) NOT NULL,
    current_node VARCHAR(64) NOT NULL,
    state_json JSONB NOT NULL DEFAULT '{}'::jsonb,
    last_error TEXT,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX IF NOT EXISTS idx_query_sessions_user_status
    ON query_sessions(user_id, status);
CREATE INDEX IF NOT EXISTS idx_query_sessions_conversation
    ON query_sessions(conversation_id);
CREATE INDEX IF NOT EXISTS idx_query_sessions_updated
    ON query_sessions(updated_at DESC);

CREATE TABLE IF NOT EXISTS draft_actions (
    action_id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    query_id UUID NOT NULL REFERENCES query_sessions(query_id) ON DELETE CASCADE,
    draft_id UUID,
    draft_version INT,
    action_type VARCHAR(64) NOT NULL,
    actor_type VARCHAR(32) NOT NULL,
    actor_id VARCHAR(128) NOT NULL,
    payload_json JSONB NOT NULL DEFAULT '{}'::jsonb,
    idempotency_key VARCHAR(128) NOT NULL,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
);

CREATE UNIQUE INDEX IF NOT EXISTS idx_draft_actions_query_idempotency
    ON draft_actions(query_id, idempotency_key);
CREATE INDEX IF NOT EXISTS idx_draft_actions_query_created
    ON draft_actions(query_id, created_at DESC);

CREATE TABLE IF NOT EXISTS learning_events (
    event_id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    event_key VARCHAR(128) NOT NULL,
    query_id UUID REFERENCES query_sessions(query_id) ON DELETE SET NULL,
    conversation_id UUID REFERENCES conversations(conversation_id) ON DELETE SET NULL,
    user_id UUID REFERENCES users(user_id) ON DELETE SET NULL,
    event_type VARCHAR(64) NOT NULL,
    event_version INT NOT NULL DEFAULT 1,
    payload_json JSONB NOT NULL DEFAULT '{}'::jsonb,
    source_component VARCHAR(128) NOT NULL,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
);

CREATE UNIQUE INDEX IF NOT EXISTS idx_learning_events_event_key
    ON learning_events(event_key);
CREATE INDEX IF NOT EXISTS idx_learning_events_query_created
    ON learning_events(query_id, created_at DESC);
CREATE INDEX IF NOT EXISTS idx_learning_events_type_created
    ON learning_events(event_type, created_at DESC);

CREATE TABLE IF NOT EXISTS governance_candidates (
    candidate_id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    candidate_type VARCHAR(64) NOT NULL,
    target_object_type VARCHAR(64) NOT NULL,
    target_object_id VARCHAR(128) NOT NULL,
    scope_type VARCHAR(64) NOT NULL,
    scope_id UUID,
    suggested_change_json JSONB NOT NULL DEFAULT '{}'::jsonb,
    evidence_summary TEXT,
    evidence_payload_json JSONB NOT NULL DEFAULT '{}'::jsonb,
    support_count INT NOT NULL DEFAULT 1,
    confidence_score NUMERIC(5,4),
    status VARCHAR(32) NOT NULL DEFAULT 'observed',
    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP,
    reviewed_at TIMESTAMP WITH TIME ZONE,
    reviewed_by UUID REFERENCES users(user_id) ON DELETE SET NULL
);

CREATE INDEX IF NOT EXISTS idx_governance_candidates_status_created
    ON governance_candidates(status, created_at DESC);
CREATE INDEX IF NOT EXISTS idx_governance_candidates_target
    ON governance_candidates(target_object_type, target_object_id);
CREATE INDEX IF NOT EXISTS idx_governance_candidates_scope
    ON governance_candidates(scope_type, scope_id);
```

#### 解析待确认阶段自然语言回复

当 `query_sessions.state_json` 中仍存在 `pending_actions` 时，前端继续提交自然语言，不应默认当成新的 `POST /api/query`。

建议先增加一个解析约定：

- `POST /api/query-sessions/{query_id}/reply-resolution`

建议请求体：

```json
{
  "query_id": "2f9f91d8-0f41-4f18-8df4-4a8c20f0f001",
  "draft_version": 2,
  "reply_text": "另外我想查今年住宅用地成交宗数",
  "conversation_id": "08a4d8cf-7247-4c2f-95b8-f2d0b3fd3001"
}
```

建议响应结果只允许为以下三类之一：

- `resolved_to_action`
  - 说明这句话是在回应当前 pending
  - 后端应把解析结果落成正式 `draft_action`
- `resolved_to_new_query`
  - 说明这句话已被判定为新问题
  - 后端应结束当前等待态，并创建新的 `query_id`
- `need_clarification`
  - 说明当前还不能安全判断
  - 前端应仅展示一轮极短澄清

建议响应体：

```json
{
  "query_id": "2f9f91d8-0f41-4f18-8df4-4a8c20f0f001",
  "resolution_status": "resolved_to_new_query",
  "resolved_action_type": "abandon_and_new_query",
  "confidence": 0.93,
  "new_query": {
    "query_id": "8e9fe5a8-3a2d-45f7-978f-1d95cb0f0003",
    "initial_text": "另外我想查今年住宅用地成交宗数"
  }
}
```

建议约束：

- 只有 `resolved_to_action` 时，才把这条自然语言回复落成当前 `query_id` 下的 `draft_actions`
- 只有 `resolved_to_new_query` 时，才允许创建新的 `query_id`
- 如果是 `need_clarification`，则不应推进主状态，也不应直接执行

#### 解析结果完成后的继续追问

当当前会话里最近一条查询已经完成，且系统识别到存在可复用结果时，用户继续输入自然语言，不应默认既不看历史结果，也不应默认总是绑定上一份结果。

建议增加一个解析约定：

- `POST /api/conversations/{conversation_id}/followup-resolution`

建议请求体：

```json
{
  "conversation_id": "08a4d8cf-7247-4c2f-95b8-f2d0b3fd3001",
  "latest_result_query_id": "2f9f91d8-0f41-4f18-8df4-4a8c20f0f001",
  "input_text": "再查去年同期，并和刚才这份结果对比"
}
```

建议响应结果只允许为以下四类之一：

- `continue_on_result`
  - 说明这句话是在继续分析上一份结果
- `compare_with_result`
  - 说明这句话是在以上一份结果为基准发起对比型追问
- `resolved_to_new_query`
  - 说明这句话是独立新问题，不应继承上一份结果上下文
- `need_clarification`
  - 说明当前还无法稳定判断是否依赖上一份结果

建议响应体：

```json
{
  "conversation_id": "08a4d8cf-7247-4c2f-95b8-f2d0b3fd3001",
  "resolution_status": "compare_with_result",
  "confidence": 0.89,
  "analysis_context": {
    "context_mode": "compare",
    "inherit_from_query_id": "2f9f91d8-0f41-4f18-8df4-4a8c20f0f001",
    "base_result_refs": [
      {
        "query_id": "2f9f91d8-0f41-4f18-8df4-4a8c20f0f001",
        "message_id": "0e0b6f72-8f2d-4b66-b2fd-91a5d4f97001",
        "result_summary": "2025年住宅用地成交面积按区域统计"
      }
    ],
    "comparison_result_refs": []
  }
}
```

建议约束：

- 只有当解析结果为 `continue_on_result` 或 `compare_with_result` 时，新的 `query_id` 才应继承 `analysis_context`
- 只有当结果为 `resolved_to_new_query` 时，新的 `query_id` 才默认不带结果上下文
- 如果结果为 `need_clarification`，则只允许补一轮极短澄清，不直接推进选表和 IR 生成

## 8. 状态、版本与兼容规则建议

### 8.1 当前已实现状态

当前最稳定的真实状态只有：

- `QueryResponse.status`
- `conversation_messages.status`
- `active_queries.status`

### 8.2 第一波核心枚举建议

### 8.3 目标态建议版本规则

建议未来统一：

- `query_drafts` 用 `version`
- `learning_events` / `system_events` 用 `event_version` 或事件类型版本
- 协议载荷结构统一使用 `schema_version`
- `plan_json` / `result_json` / `summary_json` 内若扩展子结构，也应显式带版本

### 8.4 幂等建议

建议未来遵循：

- 初始查询：`query_id`
- 用户动作：`action_id` 或 `idempotency_key`
- 学习事件：`event_key`
- 发布运行：`release_run_id`
- 评测运行：`evaluation_run_id`

### 8.5 续接与传输兼容建议

建议未来统一：

- 同一条产品级查询在多次确认、修订、继续执行过程中保持同一个 `query_id`
- `message_id` 继续代表同一条用户可见 assistant 消息
- transport 级连接标识、WebSocket 连接 ID 和重试 ID 不进入共享契约
- `original_query_id` 仅作为迁移兼容字段，不应继续作为长期主协议字段

### 8.6 第一阶段兼容建议

第一阶段仍应保留并兼容：

- `selected_table_id`
- `selected_table_ids`
- `force_execute`
- `original_query_id`

这些兼容字段只是在过渡期承接老链路，不应继续承担最终系统级对象职责。

## 9. 变更与命名约定

- 数据库字段优先使用蛇形命名
- `json/jsonb` 容器字段统一以 `_json` 结尾
- 时间字段统一以 `_at` 结尾
- 布尔字段统一以 `is_` 或 `can_` 开头
- 共享对象新增字段优先做向后兼容追加，不做无版本的破坏性重命名
- 同一对象一旦在 `05`、`07`、`15` 中出现，字段名应保持一致

## 10. 当前必须明确的边界

本文档中的目标态对象只是在统一命名和约束，不代表它们已经在代码中存在。

只要代码、初始化 SQL 和迁移里还没有这些对象，就不能把它们当成“当前系统共享真相对象”。
